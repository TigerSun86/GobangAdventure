# 敌人与奖励规则（第一版 Demo）

## 1. 文档目的与范围

本文档记录当前 **第一版 demo** 已正式采用的敌人与奖励系统规则。

本文只包含当前已经接受并进入实现范围的设计，不包含：

- 尚未进入第一版 demo 的敌人扩展方案
- 尚未进入第一版 demo 的奖励系统扩展方案
- 未来版本可能尝试但尚未验证的玩法想法

这些未进入第一版 demo 的内容应记录在：

- `Docs/game-design/tasks/deferred-gameplay-ideas.md`

本文档是当前敌人与奖励系统的 canonical gameplay 文档。若与聊天记录、临时笔记或未正式接受的草案冲突，以本文档为准。

本文同时区分两类内容：

1. **核心规则与结构**：例如敌人 HP 会跨 battle 保留、奖励必须一步完成、Skip 始终存在、同一 offer 内非 Skip 选项不能重复等。
2. **默认内容规模与默认配置值**：例如默认敌人数、默认每个敌人的 battle 上限、默认敌人固定牌组大小、默认 reward offer 规模、默认 replacement trait 数量等。这些内容在当前 demo 中有推荐默认值，但在不违反核心规则的前提下，可以通过合法 config 调整以支持测试。

若某项内容是“默认值”而非“系统硬约束”，正文中会明确写出“默认”或“推荐”的表述。

---

## 2. 敌人设计目标

当前第一版 demo 中，敌人系统的目标不是提供复杂 AI 或复杂 trait 联动，而是以较低实现成本支撑核心玩法验证。

当前敌人设计优先满足以下目标：

1. 规则简单，便于快速实现和调试。
2. 让玩家把主要注意力放在自己的牌、移位、版面和奖励成长上。
3. 通过固定牌组与基础强度梯度，提供最小但足够的敌人差异。
4. 为后续版本保留扩展敌人 trait、敌人行为模式和敌人奖励 profile 的空间。

因此，第一版 demo 的敌人复杂度会被刻意压低。

## 3. 敌人的整体结构

### 3.1 敌人数目

当前第一版 demo 的**默认内容配置**中，一局 run 包含 **3 个敌人**。

系统实际运行的敌人数目由 config 中的敌人列表决定，且敌人数至少应为 **1**。

玩家按顺序依次挑战当前 run 中的全部敌人。

### 3.2 单个敌人的 battle 上限

当前第一版 demo 的**默认内容配置**中，每个敌人的 battle 上限为 **3**。

系统实际运行时，每个敌人的 battle 上限可由其 config 单独决定，且至少应为 **1**。

每轮完整结算后：

* 若玩家最终 HP 为 0 或以下，则本局 run 立即失败；
* 只有在玩家最终 HP 大于 0 的前提下，若敌人最终 HP 为 0 或以下，才视为玩家成功击败当前敌人。

如果当前敌人的全部 battle 都已完成，并且：

* 玩家仍然存活；
* 敌人仍未被击败；

则本局 run 失败。

### 3.3 敌人 HP 的持续性

同一敌人的当前 HP 会在多场 battle 之间保留。

也就是说：

- 第 1 场 battle 对敌人造成的伤害
- 会直接影响后续 battle 开始时敌人的当前 HP

敌人的 HP 不会在 battle 之间自动回满。

### 3.4 玩家失败与敌人击败的判定优先级

玩家死亡、敌人击败与 battle 结束判定，均在一轮完整结算结束，并且本轮全部伤害与治疗结果应用完成后进行。

判定优先级为：

1. 若玩家最终 HP 为 0 或以下，则本局 run 失败；
2. 若玩家最终 HP 大于 0，且敌人最终 HP 为 0 或以下，则当前敌人被击败；
3. 若双方都仍然存活，则根据当前轮数决定继续本场 battle，或在第 3 轮后正常结束本场 battle。

若玩家与敌人在同一轮完整结算后同时降至 0 或以下，则：

* 玩家失败规则优先；
* 本局 run 失败；
* 不视为成功击败敌人；
* 不进入敌人击败、下一敌人或一局胜利流程。

当前 demo 不设置双方同时死亡后的平局、复活或特殊结局。

## 4. 敌人的牌组规则

### 4.1 敌人的固定牌组规模

当前第一版 demo 的**默认敌人固定牌组规模**为 **6 张**。

系统实际运行时，敌人的固定牌组规模由 config 中该敌人的 `fixedDeck` 内容决定。

由于当前 battle 固定为 **3 轮**，因此敌人的固定牌组规模至少应为 **3 张**，以保证每场 battle 都能提供完整的 3 张出牌序列。

当前 demo 中，敌人牌组：

- 不会在 run 中成长
- 不会在 battle 之间被永久修改
- 不会使用复杂的抽牌、弃牌或替换系统

### 4.2 当前 demo 中敌人牌没有 trait

为了降低实现复杂度与玩家认知负担，第一版 demo 中敌人牌：

- **没有 trait**
- 只包含基础类型与基础强度

这意味着敌人的战斗复杂度主要来自：

- 石头 / 剪刀 / 布 的类型关系
- 敌人牌基础强度的差异
- 每场 battle 的随机出牌顺序

### 4.3 每场 battle 的敌人出牌方式

每场 battle 开始时：

1. 对该敌人的固定牌组进行洗牌
2. 取洗牌后的前 3 张作为本场 battle 的出牌顺序
3. 第 1、2、3 轮分别使用这 3 张牌

因此在第一版 demo 中：

- 敌人没有实时 AI 决策
- 敌人不会读取玩家行为进行临场反应
- 敌人每场 battle 的差异来自固定牌组洗牌后的随机顺序

## 5. 当前 demo 的默认敌人强度模板

当前第一版 demo 的**默认内容配置**中，使用 3 个敌人的递增强度模板：

| 敌人 | 默认最大 HP | 默认固定牌组强度档位 | 默认设计意图 |
|---|---:|---|---|
| Enemy 1 | 18 | 4 / 5 | 作为最低难度敌人，用于让玩家理解基础循环与第一次奖励 |
| Enemy 2 | 24 | 5 / 6 | 提升中盘压力，要求玩家开始依赖成长后的牌组 |
| Enemy 3 | 30 | 6 / 7 | 作为当前 demo 的最终敌人，测试 build 是否足以支撑完整 run |

### 5.1 说明

- 上表是当前 demo 的默认敌人配置基线。
- 实际测试时，敌人数、敌人最大 HP、敌人固定牌组规模，以及敌人牌组中的具体强度分布，都可以通过 config 调整。
- 当前文档描述的是默认强度档位模板，不要求在玩法文档中固定每张牌的精确排列顺序。精确配置可在 config 文档和内容文件中维护。

## 6. 奖励系统设计目标

当前第一版 demo 的奖励系统设计目标如下：

1. 每次奖励都应保持小而可读。
2. 每个奖励选项都必须是**一步完成**的动作。
3. 奖励必须尽量提供有效选择，而不是形式上很多但结果重复的伪多样性。
4. 奖励系统应支持 build 成长，但不能引入过重的 UI 或二次选择流程。
5. 奖励实现方式应方便程序测试与调试。

基于这些目标，当前 demo 的奖励系统有几个核心原则：

- reward offer 默认采用小规模 4 选项结构，实际总选项数可由合法 config 调整
- 永远保留 Skip
- Upgrade 优先，Replace 作为补位
- 不允许同一 offer 出现功能等价的重复非 Skip 选项

---

## 7. 每个敌人的奖励节奏

### 7.1 每个敌人的奖励总量

当前第一版 demo 的**默认内容配置**中，每个敌人的 battle 上限为 **3**，因此默认也会提供 **3 次奖励**。

更一般地说，当前规则下：

* 每个敌人的奖励总量默认等于该敌人的 battle 上限。

这里的奖励总量是该敌人阶段在玩家存活并正常推进时可以结算的成长额度。

玩家死亡导致一局失败时，不再继续结算这些奖励额度。

### 7.2 玩家存活且敌人未被击败

如果一场 battle 在第 3 轮结束后：

* 玩家最终 HP 大于 0；
* 敌人最终 HP 大于 0；

则本场 battle 正常结束，并获得该场 battle 对应的 1 次奖励。

该次奖励结算完成后：

* 若当前敌人仍有剩余 battle，则玩家可以继续挑战当前敌人；
* 若当前敌人的 battle 上限已经耗尽，则本局 run 失败。

因此，“完成一场 battle 获得一次奖励”只适用于玩家存活并正常完成该场 battle 的情况。

### 7.3 玩家存活并击败敌人

如果一轮完整结算后：

* 玩家最终 HP 大于 0；
* 敌人最终 HP 为 0 或以下；

则当前敌人被击败。

如果玩家在该敌人的全部 battle 用完之前就击败当前敌人，那么该敌人尚未发放的剩余奖励会**立即结清**。

例如，在默认 battle 上限为 3 的情况下：

* 若第 2 场击败敌人，则在该次击败后立即结清该敌人的剩余奖励；
* 若第 1 场就击败敌人，则在该次击败后立即结清其余奖励。

若某个敌人的 battle 上限通过 config 被调整为其他数值，则其奖励总量也随之同步变化。

“立即结清”不是额外奖励，而是提前发放原本属于该敌人阶段的固定成长额度。

### 7.4 最终敌人的剩余奖励处理

如果玩家存活并提前击败最终敌人，则该敌人尚未结算的剩余奖励可以**直接忽略**。

原因是：

* run 已经胜利；
* 玩家没有后续战斗可以利用这些奖励；
* 当前 demo 不需要把最终敌人的剩余奖励转化为额外结算收益或 post-run 奖励。

### 7.5 玩家失败时的奖励处理

如果一轮完整结算后的玩家最终 HP 为 0 或以下，则本局 run 立即失败。

玩家失败后：

* 不发放导致玩家失败的该场 battle 对应奖励；
* 不结清当前敌人的剩余奖励；
* 不生成新的 reward offer；
* 不进入下一场 battle；
* 不进入下一名敌人；
* 不进入一局胜利流程。

即使敌人在同一轮完整结算后也降至 0 或以下，以上规则仍然不变：

* 玩家失败规则优先；
* 不视为成功击败敌人；
* 不发放敌人击败奖励。

## 8. Reward Offer 的默认结构

当前第一版 demo 的**默认 reward offer 总选项数**为 **4**。

当前规则下：

- 每一次 reward offer 都固定包含 **1 个 `Skip`**
- 除 `Skip` 以外的其余选项为非 Skip 选项
- 非 Skip 选项只可能来自两种类型：
  - `Upgrade`
  - `Replace`

因此，若某次 reward offer 的总选项数为 `N`，则：

- 其中固定有 1 个 `Skip`
- 其余 `N - 1` 个为非 Skip 选项

在当前第一版 demo 的默认配置中：

- reward offer 总选项数 = 4
- 因此默认结构为：
  - 3 个非 Skip
  - 1 个 Skip

当前第一版 demo 中仍不使用：

- 第 4 种奖励类型
- 玩家主动刷新按钮
- 二次目标选择流程

---

## 9. Upgrade / Replace 的组成规则

### 9.1 基本原则

在当前规则下：

- `Skip` 固定为 1 个
- `Replace` 的数量不单独配置
- `Replace` 的职责是补齐剩余的非 Skip 槽位

也就是说，当前 reward offer 的组成由以下因素共同决定：

- reward offer 的总选项数
- 固定的 1 个 `Skip`
- 当前允许的 `Upgrade` 目标数量
- 当前实际可生成的合法 `Upgrade` 数量

### 9.2 当前默认配置

当前第一版 demo 的**默认配置**为：

- reward offer 总选项数：4
- `Skip` 数量：固定为 1
- `Upgrade` 的默认上限：2

因此在默认情况下，reward offer 的非 Skip 槽位总数为：

- `4 - 1 = 3`

在这 3 个非 Skip 槽位中，系统会先尽量生成合法且互不重复的 `Upgrade`，剩余位置再由 `Replace` 自动补齐。

### 9.3 当前默认配置下的典型组合

在默认配置下，reward offer 可能呈现为：

- `2 Upgrade + 1 Replace + 1 Skip`
- `1 Upgrade + 2 Replace + 1 Skip`
- `0 Upgrade + 3 Replace + 1 Skip`

### 9.4 一般化规则

若当前配置下：

- reward offer 总选项数为 `N`
- `Skip` 固定为 1
- `Upgrade` 目标上限为 `U`

则：

- 非 Skip 槽位总数 = `N - 1`
- `Upgrade` 的目标数量至多为 `min(U, N - 1)`
- 系统会先生成合法且互不重复的 `Upgrade`
- 若实际可生成的 `Upgrade` 少于目标数量，则剩余非 Skip 槽位由 `Replace` 自动补齐

### 9.5 配置合法性边界

若 reward offer 的总选项数为 `N`，则：

- `N` 至少应为 **1**
- `Upgrade` 目标上限至少可为 **0**
- `Upgrade` 目标上限至多为 `N - 1`

这意味着在测试配置中，允许出现：

- 只有 `Skip` 的 reward offer
- 全部非 Skip 都由 `Replace` 补齐的 reward offer
- 非常偏向 `Upgrade` 的 reward offer

### 9.6 同一 offer 内禁止重复非 Skip 选项

在同一次 reward offer 中：

- 不允许重复的 Upgrade 选项
- 不允许重复的 Replace 选项
- 不允许出现功能等价的非 Skip 选项

具体的“重复”判定规则见本文第 13 节。

## 10. Upgrade 规则（玩法层）

### 10.1 Upgrade 的含义

`Upgrade` 的含义是：

- 对当前牌组中的某一张现有牌
- 增加 1 个新的 trait

它是对现有牌的**原地增强**。

### 10.2 Upgrade 的基本约束

当前 demo 中，Upgrade 必须满足：

- 目标牌当前 trait 数少于 3
- 不能加入已存在的 trait
- 不能使同一张牌同时拥有左移位与右移位

### 10.3 当前 demo 中 Upgrade 不直接提高基础强度

当前第一版 demo 中，Upgrade 的作用范围是：

- **增加新 trait**

而不是：

- 直接提高基础强度

这样做的原因是：

1. 避免与“强化”trait 的定位重叠。
2. 简化数值设计与 reward 理解成本。
3. 让 reward 的成长更多体现在功能扩展和组合变化上。

### 10.4 Upgrade 必须一步完成

玩家不需要先选“升级”，再选目标牌。

当前 demo 中，每个 Upgrade 选项本身必须已经绑定好：

- 要加到哪张牌
- 要加哪个 trait

也就是说，Upgrade 是一步完成的直接动作。

---

## 11. Replace 规则（玩法层）

### 11.1 Replace 的含义

`Replace` 的含义是：

- 用一张新牌
- 直接替换当前牌组中的一张现有牌

Replace 是真正的替换，而不是修改原牌的一个属性。

### 11.2 Replace 必须一步完成

玩家不需要先选一张新牌，再选要替换哪张旧牌。

在当前 demo 中，每个 Replace 选项本身必须已经绑定好：

- 要替换哪张旧牌
- 替换成什么新牌

因此 Replace 和 Upgrade 一样，都是**一步完成**的奖励动作。

### 11.3 当前 demo 中 Replace 新牌的生成原则

Replace 使用的新牌不是从“完全手工列好的固定大卡池”中取出，而是：

- 根据合法生成规则动态生成

当前第一版 demo 中，对 replacement card 的生成规则为：

- 使用一个固定配置的 base power
- trait 数量由 config 决定
- trait 组合必须合法

虽然未来架构可以支持多个可选 base power，但在当前 demo 中，玩家 replacement card 的 base power 统一使用一个固定配置值。

### 11.4 Replace 新牌的 trait 数量

当前第一版 demo 的**默认 replacement trait 数量**为 **2**。

更一般地说：

- replacement card 的 trait 数量由 config 决定
- 当前允许的测试范围为 **0 到 3**
- 当前默认推荐值仍为 **2**

当 replacement trait 数量大于 1 时，trait 之间仍需满足当前牌面合法性要求，例如：

- 不能重复
- 不能同时包含 **左移位** 与 **右移位**

因此，在测试配置中，允许出现：

- 没有 trait 的 replacement card
- 只有 1 个 trait 的 replacement card
- 2 trait 或 3 trait 的 replacement card

### 11.5 Replace 允许跨基础类型

Replace 不要求新牌与被替换牌拥有相同基础类型。

这意味着玩家可以在 run 中逐步形成偏科构筑，例如：

- 更多石头牌
- 更少布牌
- 集中某一基础类型的打法

当前第一版 demo 不对同类型牌数量设置上限。

---

## 12. Skip 规则

`Skip` 是 reward offer 中固定存在的正式选项。

### 12.1 Skip 的效果

- 不改变当前牌组
- 不改变当前牌的 trait
- 不改变当前牌的基础强度

### 12.2 为什么 Skip 必须存在

Skip 的存在有几个作用：

1. 防止玩家被迫接受自己不喜欢的奖励。
2. 在当前牌组已经形成清晰 build 时，允许玩家保持结构不变。
3. 避免 reward 选项虽然都“合法”，但对当前 build 都不合适时产生糟糕体验。

### 12.3 Skip 不是异常兜底

Skip 是当前设计中的正式选项，而不是当奖励生成失败时才补上的异常兜底机制。

---

## 13. 奖励去重规则（玩法层表述）

当前第一版 demo 中，同一 reward offer 内的非 Skip 选项不能重复。

这里的“重复”不是仅指文案或表面动作完全一样，而是指：

> 若两个不同的奖励选项在应用后会得到**功能等价的最终牌组结果**，则它们应视为重复，不能同时出现。

### 13.1 判定功能等价时忽略的内容

在判断两个奖励选项是否导向相同结果时，忽略以下差异：

- 卡实例身份
- 牌组顺序
- trait 顺序

### 13.2 例子

若出现以下两种 Upgrade：

- 选项 A：把卡 X 从 `[A]` 升级到 `[A, B]`
- 选项 B：把卡 Y 从 `[B]` 升级到 `[A, B]`

如果 X 与 Y 除 trait 顺序差异外在功能上完全等价，则这两个选项应被视为重复，不应同时出现在同一 reward offer 中。

### 13.3 设计目的

这一规则的目的，是避免 reward offer 给玩家展示“表面上不同、实际上结果一样”的伪选择。

奖励系统希望提供的是：

- 真正不同的 build 方向
- 真正不同的 deck 结果

而不是多个换皮后的等价选项。

## 14. 当前奖励系统明确不包含的内容

为了快速完成第一版 demo，当前奖励系统明确不包含以下内容：

- 玩家主动刷新 reward offer
- 多步式升级或替换流程
- 商店 / 金币购买
- 更复杂的 weighted reward profile
- replacement 使用多个 base power 混搭
- 基于稀有度的复杂池管理
- 跨 reward 的长期保底机制

这些内容如后续需要，应先进入 deferred 文档，再决定是否纳入正式设计。

## 17. 小结

当前第一版 demo 的敌人与奖励系统，采用的是一种**刻意压低复杂度但保留关键策略性的设计**。

敌人方面：

- 当前默认配置下，敌人使用较小规模的固定牌组
- 没有 trait
- 没有复杂 AI
- 通过 HP、牌组规模与基础强度档位提供最小梯度
- 在不违反核心规则的前提下，敌人数、每个敌人的 battle 上限、敌人固定牌组规模等内容可通过 config 调整
- 每轮完整结算后检查玩家生存状态
- 玩家最终 HP 为 0 或以下时，本局立即失败
- 双方同时降至 0 或以下时，玩家失败规则优先

奖励方面：

- 当前默认配置下，reward offer 采用小规模结构
- `Skip` 固定存在
- `Upgrade` 优先，`Replace` 用于补齐剩余非 Skip 槽位
- 强调一步完成、避免重复、避免伪多样性
- 在不违反核心规则的前提下，reward offer 总选项数、Upgrade 目标上限、replacement trait 数量等内容可通过 config 调整
- 玩家失败后不生成本场奖励，也不结清当前敌人的剩余奖励

这套设计的目的，是在第一版 demo 中以最低必要复杂度，支撑对核心玩法循环的有效验证，同时保留少量配置层测试空间。
