# 游戏规则（已锁定）

## 1. 文档目的与适用范围

本文档记录当前 **第一版 demo** 已接受并锁定的核心玩法规则。

本文只包含当前实现应遵循的正式玩法规则，不包含尚未进入第一版 demo 的候补想法、延期内容或未来扩展方案。此类内容应单独记录在：

- `Docs/game-design/tasks/deferred-gameplay-ideas.md`

本文是当前玩法层面的主规则文档。程序设计、程序实现与测试都应以本文为当前有效玩法规则的上位参考。

本文同时区分两类内容：

1. **核心不变量**：例如 3 槽位结构、3 轮 battle、石头 / 剪刀 / 布基础类型、固定轮内结算顺序等。这些属于当前 demo 的锁定规则。
2. **默认内容规模与默认配置值**：例如默认敌人数、默认每个敌人的 battle 上限、默认起始牌组规模、默认奖励报价规模等。这些内容在当前 demo 中有推荐默认值，但在不违反核心不变量的前提下，可以通过合法 config 调整以支持测试。

若某项内容是“默认值”而非“系统硬约束”，正文中会明确写出“默认”或“推荐”的表述。

---

## 2. 游戏定位与设计目标

本项目是一款 **短流程、回合制、卡牌式、石头剪刀布 roguelike demo**。

当前阶段的目标不是完成正式商业版本，而是以较低实现成本快速验证以下核心问题：

1. 三轮累积结算是否有趣。
2. 移位机制是否能够带来明确的策略性。
3. 同类型但不同功能的牌之间是否存在真实取舍。
4. 小牌库成长是否能够形成可感知的 build。

当前第一版 demo 明确优先：

- 正确性
- 规则清晰度
- 可调试性
- 快速迭代

当前第一版 demo 不优先：

- 最终美术表现
- 最终正式 UI
- 完整 meta 系统
- 大规模内容量

---

## 3. 一局 Run 的总体流程

当前第一版 demo 的**默认内容配置**中，一局 run 由 **3 个敌人** 组成，玩家按顺序依次挑战。

系统实际运行的敌人数量由 config 中的敌人列表决定，且敌人数至少应为 **1**。

每个敌人：

* 当前第一版 demo 的**默认 battle 上限**为 **3 场**。
* 实际每个敌人的 battle 上限可由 config 单独决定，且至少应为 **1**。
* 每轮完整结算后，若玩家最终 HP 为 0 或以下，则本局 run 立即失败。
* 只有在玩家最终 HP 大于 0 的前提下，若敌人最终 HP 为 0 或以下，才视为玩家成功击败当前敌人。
* 若当前敌人的全部 battle 都打完后，玩家仍然存活但敌人尚未被击败，则本局 run 失败。

当玩家存活并成功击败当前 run 中的最后一个敌人后，本局 run 胜利。

若玩家与敌人在同一轮结算后同时降至 0 或以下，则玩家失败规则优先，本局 run 失败。

---

## 4. 玩家与敌人的持久状态

### 4.1 玩家状态

玩家拥有：

- 当前 HP
- 最大 HP
- 当前牌组

当前第一版 demo 中，玩家起始当前 HP 与最大 HP 均为 **30**。

也就是说：

- Run 开始时，玩家当前 HP = 30
- Run 开始时，玩家最大 HP = 30

玩家的当前 HP 会在不同敌人之间保留，不会在进入新敌人时自动回满。

当前 demo 中，玩家最大 HP 在一局 run 内固定，不存在提升最大 HP 上限的正式机制。

### 4.2 敌人状态

每个敌人在与玩家连续交战的过程中拥有：

- 当前 HP
- 最大 HP
- 当前已进行的 battle 数
- 当前已结算的奖励数

敌人的当前 HP 会在与该敌人的多场 battle 之间保留。

### 4.3 治疗上限规则

所有治疗与吸血效果都不能使当前 HP 超过最大 HP。

该规则对玩家与敌人都成立。

### 4.4 玩家生存判定

玩家是否存活，不在合并伤害刚刚扣除时立即判定，而是在本轮完整结算结束后判定。

也就是说：

* 本轮先正常应用合并伤害；
* 再正常处理结算后阶段中的治疗、吸血与其他效果；
* 等本轮全部伤害与治疗结果应用完成后，再检查玩家最终 HP。

因此，玩家 HP 可以在合并伤害应用后暂时降至 0 或以下，并通过本轮结算后效果恢复到 0 以上。

若本轮完整结算后的玩家最终 HP 仍为 0 或以下，则本局 run 立即失败。

---

## 5. 玩家牌组的基础规则

### 5.1 起始牌组规模

当前第一版 demo 的**默认起始牌组规模**为 **6 张**。

实际起始牌组规模由 config 中的 `startingDeck` 内容决定。

由于当前 battle 固定为 **3 轮**，且同一张牌在同一场 battle 中最多使用 1 次，因此起始牌组规模至少应为 **3 张**。

当前第一版 demo 中不使用额外抽牌、洗牌、弃牌、战前编组等系统。

### 5.2 Battle 中的用牌规则

每场 battle：

- 玩家从**当前牌组**中选择出牌。
- 每场 battle 一共进行 3 轮，因此玩家本场 battle 最多打出 3 张牌。
- **同一张牌在同一场 battle 中最多使用 1 次。**

### 5.3 单牌 trait 规则

每张牌：

- 最多拥有 **3 个 trait**。
- 不能拥有重复 trait。
- 不能同时拥有 **左移位** 与 **右移位**。

---

## 6. 默认起始牌组配置

当前第一版 demo 的**默认起始配置**采用方案 A：

1. 石头【强化】，基础强度 4
2. 石头【左移位】，基础强度 4
3. 剪刀【强化】，基础强度 4
4. 剪刀【左移位】，基础强度 4
5. 布【强化】，基础强度 4
6. 布【左移位】，基础强度 4

这样设计的目的，是在第一场 battle 中就让玩家体验到：

- 纯数值牌与功能牌的差异
- 同基础类型但不同功能牌之间的取舍
- 移位机制对版面和结算结果的影响

以上内容是当前 demo 的默认起始牌组基线，而不是系统唯一允许的起始牌组形式。

在不违反当前核心规则的前提下，测试时可以通过 config 调整起始牌组的具体内容与牌组规模。

---

## 7. 单场 Battle 的结构

每场 battle 固定进行 **3 轮**。

每轮中：

- 玩家打出 1 张牌
- 敌人同时打出 1 张牌
- 双方的新牌分别进入当前轮次对应的新位置

轮次与新位置的关系为：

- 第 1 轮：进入位置 1
- 第 2 轮：进入位置 2
- 第 3 轮：进入位置 3

进入场上的牌不会在回合结束后离场，而是会一直留在当前 battle 的版面上，直到本场 battle 结束。

---

## 8. 位置与版面规则

### 8.1 三个位置

每个 battle 的玩家侧和敌人侧都各有 3 个位置：

- 位置 1
- 位置 2
- 位置 3

### 8.2 位置开启顺序

位置会随着轮次逐步开启：

- 第 1 轮只开启位置 1
- 第 2 轮开启位置 1、2
- 第 3 轮开启位置 1、2、3

### 8.3 位置价值

因为每轮都会重新结算所有已开启位置，因此：

- 位置 1 会被结算 **3 次**
- 位置 2 会被结算 **2 次**
- 位置 3 会被结算 **1 次**

这使得位置 1 的价值天然更高，但移位机制可以在后续轮次中改写版面，从而改变早前位置的意义。

### 8.4 位置以当前槽位为准

一张牌进入场上后，之后所有效果与结算都以它当前所在的槽位为准，而不是以“它是第几轮打出来的牌”为准。

若牌因为移位而改变位置，则其新的位置就是当前真实版面状态。

---

## 9. 每轮结算的高层顺序

每轮按以下顺序结算：

1. 入场
2. 固定自属性
3. 移位
4. 版面派生
5. 结算当前所有已开启位置
6. 合并伤害并同时扣血
7. 结算后效果

这一顺序属于当前 demo 的锁定玩法规则，不应在实现中随意调整。

七个轮内结算阶段必须完整执行。

玩家或敌人的 HP 在合并伤害阶段后暂时降至 0 或以下，不会中断本轮尚未完成的结算后效果。

当结算后阶段完成，并且本轮全部伤害与治疗结果均已应用后，系统才进行玩家生存、敌人击败与 battle 结束判定。

该判定发生在七阶段轮内结算完成之后，不属于新的轮内结算阶段。

---

## 10. 胜负与伤害规则

### 10.1 胜负关系

每张牌有一个基础类型：

- 石头
- 剪刀
- 布

胜负关系遵循标准石头剪刀布关系：

- 石头胜剪刀
- 剪刀胜布
- 布胜石头

### 10.2 平局规则

若双方同一位置基础类型相同，则视为平局。

平局时：

- 双方都不受伤害
- 忽略双方强度差

### 10.3 伤害公式

若一方在某位置获胜，则该位置造成的伤害为：

`max(1, 胜者本轮强度 - 败者本轮强度)`

也就是说：

- 若强度差较大，则造成更多伤害
- 若胜者强度没有高过败者，也至少造成 1 点保底伤害

### 10.4 同时结算规则

本轮所有已开启位置的伤害先分别累计，之后再同时应用到双方 HP。

不会按位置逐个立即扣血。

---

## 11. 移位规则

### 11.1 左移位

若一张牌具有 **左移位**，则在移位阶段中，当系统检查到它时：

- 如果其左侧有友方牌，则与左侧友方牌交换位置

### 11.2 右移位

若一张牌具有 **右移位**，则在移位阶段中，当系统检查到它时：

- 如果其右侧有友方牌，则与右侧友方牌交换位置

### 11.3 检查顺序

移位阶段的检查顺序为：

- 从右到左
- 依次检查位置 3、位置 2、位置 1

### 11.4 版面持续性

移位后的结果会保留到下一轮。

也就是说：

- 一轮结束后，牌不会自动回到原位
- 下一轮开始时，玩家看到的就是上一轮结束后的真实版面状态

这一点是当前策略性的关键来源之一。

---

## 12. 单场 Battle 的结束条件

### 12.1 先完成本轮全部结算

每轮必须先完整执行当前锁定的七个轮内结算阶段：

1. 入场
2. 固定自属性
3. 移位
4. 版面派生
5. 结算当前所有已开启位置
6. 合并伤害并同时扣血
7. 结算后效果

在本轮全部伤害、治疗与结算后效果应用完成之前，不进行玩家死亡、敌人击败或 battle 结束判定。

因此：

* 玩家 HP 在合并伤害应用后暂时降至 0 或以下，不会立刻中断本轮；
* 玩家仍然可以通过本轮的回春、吸血或其他结算后治疗恢复 HP；
* 最终判定只读取本轮完整结算后的最终 HP。

### 12.2 轮末判定优先级

本轮完整结算后，按以下优先级进行判定：

1. 若玩家最终 HP 为 0 或以下，则本局 run 立即失败。
2. 若玩家最终 HP 大于 0，且敌人最终 HP 为 0 或以下，则当前敌人被击败。
3. 若玩家与敌人都仍然存活，且第 3 轮结算完成，则本场 battle 正常结束。
4. 若玩家与敌人都仍然存活，且尚未完成第 3 轮，则进入下一轮。

### 12.3 双方同时降至 0 或以下

若玩家与敌人在同一轮完整结算后同时降至 0 或以下，则：

* 玩家失败规则优先；
* 本局 run 失败；
* 不视为一次能够推动 run 继续进行的成功敌人击败；
* 不进入敌人击败后的奖励、下一敌人或胜利流程。

当前 demo 不设置双方同时死亡后的平局、复活或特殊结局。

### 12.4 Battle 上限失败

若一场 battle 在第 3 轮结束时：

* 玩家仍然存活；
* 敌人仍然存活；

则本场 battle 正常结束。

之后：

* 若当前敌人仍有剩余 battle，则按照正常奖励流程结算后，可以继续挑战该敌人；
* 若当前敌人的 battle 上限已经耗尽，则本局 run 失败。

因此，一局 run 当前存在两类正式失败条件：

1. 玩家在一轮完整结算后的最终 HP 为 0 或以下；
2. 玩家在当前敌人的 battle 上限内未能击败该敌人。

## 13. 奖励发放总规则

### 13.1 每个敌人的奖励总量

当前第一版 demo 的**默认内容配置**中，每个敌人的 battle 上限为 **3**，因此默认也会提供 **3 次奖励额度**。

更一般地说，当前规则下：

- 每个敌人的奖励总量默认等于该敌人的 battle 上限。

### 13.2 正常奖励节奏

通常情况下：

- 每完成 1 场 battle，获得 1 次奖励

### 13.3 提前击败时的奖励结清

如果敌人在用完全部 battle 之前被击败，则该敌人剩余未发放的奖励会立即结清。

例如，在默认 battle 上限为 3 的情况下：

- 若第 2 场击败敌人，则击败后立即获得该敌人剩余的奖励
- 若第 1 场击败敌人，则击败后立即获得该敌人剩余的奖励

若某个敌人的 battle 上限通过 config 被调整为其他数值，则其奖励总量也随之同步变化。

### 13.4 最终敌人的特殊规则

当当前 run 中的最终敌人被击败后：

- 若该敌人仍有未结清奖励，则这些剩余奖励 **直接忽略**
- 玩家直接进入胜利结算

### 13.5 玩家失败时的奖励处理

若玩家在一轮完整结算后的最终 HP 为 0 或以下，则本局 run 立即失败。

玩家失败后：

* 不发放导致玩家失败的该场 battle 对应奖励；
* 不结清当前敌人的剩余奖励；
* 不生成新的 reward offer；
* 不进入下一场 battle；
* 不进入下一名敌人；
* 即使敌人在同一轮也降至 0 或以下，仍然不进入敌人击败后的奖励或推进流程。

## 14. 当前 demo 明确不包含的内容

以下内容不属于当前第一版 demo 的正式玩法范围：

- 商店 / 金币系统
- 无尽模式
- 最大 HP 上限提升
- 更复杂的敌人 AI 决策
- 敌人复杂 trait 系统
- 跨战斗永久禁用敌方牌库的正式系统
- 战前编组系统
- 抽牌 / 洗牌 / 弃牌系统
- 多语言支持

如未来重新考虑这些内容，应先进入 deferred 或 proposal 文档，再决定是否提升为正式规则。
