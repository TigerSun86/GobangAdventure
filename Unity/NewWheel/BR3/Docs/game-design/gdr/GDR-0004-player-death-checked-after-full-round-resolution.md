# GDR-0004: Player Death Checked After Full Round Resolution

## Status

Accepted

---

## Context

本文记录第一版 demo 中“玩家死亡在何时判定，以及双方同时降至 0 或以下时如何处理”的关键玩法决策。

当前战斗采用固定的七阶段轮内结算流程：

1. 入场阶段
2. 固定自属性阶段
3. 移位阶段
4. 版面派生阶段
5. 槽位结算阶段
6. 合并伤害阶段
7. 结算后阶段

当前正式卡牌特质中：

* 回春会在结算后阶段回复玩家 HP；
* 吸血会在结算后阶段根据该牌本轮造成的伤害回复玩家 HP；
* 成长也会在结算后阶段处理。

与此同时，槽位伤害会先汇总，再在合并伤害阶段作为本轮结果同时应用。

因此，当玩家 HP 在合并伤害阶段降至 0 或以下时，会出现一个必须明确的问题：

> 玩家是否立刻死亡，从而跳过本轮尚未处理的回春、吸血与其他结算后效果？

还需要处理另一个与同时结算直接相关的问题：

> 如果玩家与敌人在同一轮完整结算后同时降至 0 或以下，这一局应当视为玩家获胜、玩家失败，还是特殊平局？

旧规则没有将玩家 HP 归零作为正式失败条件，而只把“未能在当前敌人的 battle 上限内击败敌人”视为一局失败。

该规则会削弱玩家 HP、治疗、吸血、续航和未来低血构筑的意义，因此需要正式修正，并单独记录决策理由。

---

## Decision

本次接受的决策是：

* 每轮必须完整执行当前锁定的七个轮内结算阶段；
* 玩家或敌人的 HP 在合并伤害阶段后暂时降至 0 或以下，不会中断本轮尚未完成的结算后效果；
* 玩家死亡、敌人被击败与 battle 结束判定，在结算后阶段完成，并且本轮全部伤害与治疗结果应用完毕后进行；
* 若玩家最终 HP 为 0 或以下，则本局 run 立即失败；
* 若玩家最终 HP 大于 0，并且敌人最终 HP 为 0 或以下，则当前敌人被击败；
* 若玩家与敌人在同一轮完整结算后同时降至 0 或以下，则玩家死亡规则优先，本局 run 失败；
* 玩家死亡后不发放导致死亡的该场 battle 对应奖励，也不结清当前敌人的剩余奖励；
* 轮末结果判定不属于新的第八个轮内结算阶段。

在当前 v1 demo 中，此规则视为锁定规则。

---

## Detailed Decision

### 1. 固定七阶段必须完整执行

当前七阶段轮内结算顺序保持不变：

```text
Enter
→ Fixed Self
→ Movement
→ Board Derived
→ Resolve Open Slots
→ Apply Merged Damage
→ Post Resolve
```

在进入结算后阶段之前，不进行玩家死亡、敌人被击败或 battle 结束判定。

在结算后阶段尚未完成时，也不提前中断本轮。

这样可以确保：

* 回春正常处理；
* 吸血正常处理；
* 成长正常处理；
* 后续属于结算后阶段的正式效果拥有稳定的执行位置。

### 2. HP 可以在轮内暂时降至 0 或以下

合并伤害应用后，玩家 HP 可以暂时为：

* `0`
* 负数

这时玩家尚未被正式判定死亡。

如果本轮结算后阶段存在合法治疗，则继续应用这些治疗。

例如：

```text
玩家轮前 HP：2
本轮受到伤害：3
本轮回春治疗：2

最终 HP：
2 - 3 + 2 = 1
```

玩家最终 HP 为 1，因此玩家存活。

这里的暂时归零只是一轮计算中的中间状态，不是最终死亡结果。

### 3. 最终 HP

本规则中的“最终 HP”指：

> 本轮七个轮内结算阶段全部完成，并且本轮全部伤害、治疗与其他 HP 变化应用后的 HP。

只有最终 HP 用于判断：

* 玩家是否死亡；
* 敌人是否被击败；
* battle 是否继续；
* battle 是否结束；
* run 是否进入失败、奖励或后续敌人流程。

### 4. 轮末判定优先级

本轮完整结算后，按以下顺序判定：

1. 玩家死亡；
2. 敌人被击败；
3. battle 是否达到第 3 轮；
4. battle 是否继续。

具体规则为：

#### 玩家最终 HP 为 0 或以下

* 玩家死亡；
* 本局 run 失败；
* 不再继续检查能够推动 run 前进的敌人击败结果；
* 不进入奖励流程。

#### 玩家最终 HP 大于 0，敌人最终 HP 为 0 或以下

* 当前敌人被击败；
* 按正常规则处理奖励、下一敌人或一局胜利。

#### 玩家与敌人都仍然存活，且第 3 轮已经完成

* 当前 battle 正常结束；
* 之后由 run 流程判断是否仍有剩余 battle，或是否因耗尽 battle 上限而失败。

#### 玩家与敌人都仍然存活，且尚未完成第 3 轮

* 进入下一轮。

### 5. 双方同时降至 0 或以下

若玩家与敌人在同一轮完整结算后同时降至 0 或以下，则：

* 玩家死亡规则优先；
* 本局 run 失败；
* 不视为一次成功击败敌人；
* 不进入敌人击败后的奖励流程；
* 不进入下一敌人；
* 即使当前敌人是最终敌人，也不进入一局胜利。

当前 demo 不设置：

* 同归于尽也算玩家获胜；
* 玩家以 0 HP 进入下一敌人；
* 同时死亡后的复活；
* 同时死亡后的特殊平局；
* 同时死亡后的特殊奖励。

### 6. 玩家死亡后的奖励

玩家死亡导致一局失败后：

* 不发放导致死亡的该场 battle 对应奖励；
* 不结清当前敌人的剩余奖励；
* 不生成新的 reward offer；
* 不进入下一场 battle；
* 不进入下一敌人；
* 不进入一局胜利。

该规则同样适用于双方同时降至 0 或以下的情况。

---

## Rationale

## 1. 保持七阶段流程完整

当前轮内结算顺序是核心规则，不应因为 HP 临时归零而在中间插入特殊中断。

如果在合并伤害应用后立刻停止：

* 结算后阶段可能无法完整执行；
* 回春和吸血的触发规则会出现例外；
* 后续新增结算后效果时需要逐个判断是否允许在“致命伤害后”执行；
* round pipeline 会变得难以理解和测试。

完整执行七阶段，再统一判定结果，规则更稳定。

## 2. 让回春与吸血真正具有生存意义

回春与吸血属于结算后阶段。

如果玩家在合并伤害后立即死亡，它们无法挽救本轮受到致命伤害的玩家，只能用于回复非致命伤害。

这会显著降低两种特质的玩法价值，也会让玩家难以理解：

> 为什么一个明确写在本轮结算中的治疗效果，没有机会在本轮生效？

允许结算后治疗救回玩家，可以让：

* 回春成为可靠的生存特质；
* 吸血支持高风险换血；
* 未来续航型全局修正拥有清晰基础。

## 3. 尊重伤害的同时结算原则

当前槽位伤害先汇总，再作为本轮合并结果同时应用。

因此，玩家与敌人可能同时降至 0 或以下。

不应通过：

* 槽位处理顺序；
* 某一方 HP 先写入；
* 某个对象先执行代码；

来决定谁先死亡。

双方最终 HP 应在完整轮结算后统一读取，再按照明确的玩法优先级处理。

## 4. 玩家必须存活才能取得胜利

当前基础规则要求：

> 玩家存活，并击败敌人，才能继续推进或取得一局胜利。

若双方同时死亡仍算玩家获胜，就会产生额外问题：

* 非最终敌人之后玩家以多少 HP 继续；
* 是否自动复活；
* 是否仍然发放奖励；
* 同归于尽是否比正常击败更有利；
* 最终敌人与普通敌人的处理是否不同。

玩家死亡优先可以避免这些额外规则，并保持生存资源的重要性。

## 5. 为后续玩法建立清晰基础

这项规则会成为后续设计的共同基础，包括：

* 低血流；
* 续航型全局修正；
* 治疗修正；
* 吸血联动；
* 减伤；
* 护盾；
* 受到致命伤害时触发的候选效果；
* 复活或免死机制。

在正式引入这些系统之前，必须先明确：

* 什么是存活；
* 什么是死亡；
* 什么时候判定死亡；
* 同时死亡如何处理。

---

## Alternatives Considered

## Alternative 1: 合并伤害后立即判断玩家死亡

### Summary

合并伤害应用后，只要玩家 HP 为 0 或以下，就立即结束 battle 和 run，不再处理结算后阶段。

### Pros

* 与部分传统游戏“HP 归零立刻死亡”的直觉一致；
* 流程表面上更直接；
* 可以更早停止无意义计算。

### Cons

* 回春和吸血无法挽救本轮致命伤害；
* 会在第 6 与第 7 阶段之间插入特殊终止条件；
* 破坏七阶段完整性；
* 未来所有结算后效果都必须额外处理死亡中断；
* 玩家难以理解为什么本轮已经满足条件的治疗没有生效。

### Why It Was Not Chosen

当前玩法将回春、吸血明确放在结算后阶段。完整轮结算后再判断死亡，更符合已锁定的阶段结构，也能让这些特质发挥预期作用。

---

## Alternative 2: 双方同时死亡时敌人被击败优先

### Summary

若双方同时降至 0 或以下，则仍视为玩家成功击败敌人。

### Pros

* 鼓励高风险换血；
* 同归于尽击败最终敌人可能具有戏剧性；
* 对爆发型构筑较宽容。

### Cons

* 玩家死亡不再稳定地导致失败；
* 非最终敌人之后必须定义复活或后续 HP；
* 可能让玩家主动追求同归于尽；
* 会削弱续航、防御与生存构筑的价值；
* 普通敌人与最终敌人的处理可能需要分叉。

### Why It Was Not Chosen

当前 demo 不需要额外的复活、特殊胜利或同归于尽推进规则。要求玩家存活才能继续，更简单也更一致。

---

## Alternative 3: 双方同时死亡时视为特殊平局

### Summary

双方同时降至 0 或以下时，不算胜利也不算普通失败，而进入独立的平局或特殊结局。

### Pros

* 最直接体现同时结算；
* 可以保留特殊叙事或模式设计空间。

### Cons

* 需要新增终局类别；
* 需要定义奖励、敌人推进和最终敌人处理；
* 需要新增 UI、日志、测试与流程分支；
* 对当前 demo 的验证目标没有足够价值。

### Why It Was Not Chosen

当前 demo 应保持终局规则简单，只保留一局胜利与一局失败。

---

## Alternative 4: 玩家 HP 不会导致一局失败

### Summary

保留旧规则：玩家 HP 即使降至 0 或以下也继续游戏，一局只在 battle 上限耗尽且敌人未被击败时失败。

### Pros

* 不需要新增死亡流程；
* 当前 run progression 结构最简单；
* 不会出现同时死亡问题。

### Cons

* 玩家 HP 不再是真正的生存资源；
* 可能出现负 HP 继续战斗；
* 回春、吸血、减伤和护盾价值明显降低；
* 低血与死亡没有边界；
* 敌人输出无法直接造成失败；
* 不符合玩家对 HP 的基本理解。

### Why It Was Not Chosen

该规则被确认为早期遗漏。玩家 HP 应当具有明确的失败意义，否则多个现有和未来玩法系统都缺乏可靠基础。

---

## Consequences

### Positive Consequences

* 玩家 HP 成为真正的生存资源；
* 回春与吸血可以在本轮救回玩家；
* 七阶段结算流程保持完整；
* 同时死亡拥有明确且可测试的结果；
* 低血、续航、护盾和减伤等未来设计获得稳定基础；
* 玩家死亡后奖励流程清晰终止；
* 日志和 Debug UI 可以明确区分暂时归零与最终死亡。

### Costs and Risks

* 实现需要增加玩家死亡与同时死亡测试；
* battle outcome 需要能够表达玩家死亡结果；
* run flow 必须在玩家死亡后直接进入 Defeat；
* 失败时必须确保不生成 reward offer；
* Debug UI 需要清楚显示死亡原因；
* HP 在轮内可能暂时为负数，日志和中间结果展示需要避免造成误解。

这些成本被认为是必要且可控的。

## Gameplay Clarifications

### Example 1: 回春救回玩家

```text
轮前玩家 HP：2
玩家受到伤害：3
回春治疗：2
最终玩家 HP：1
```

结果：

* 玩家存活；
* 正常继续进行轮末结果判定。

### Example 2: 治疗后最终 HP 恰好为 0

```text
轮前玩家 HP：2
玩家受到伤害：4
回春治疗：2
最终玩家 HP：0
```

结果：

* 玩家死亡；
* 本局失败。

### Example 3: 玩家与敌人同时归零

```text
最终玩家 HP：0
最终敌人 HP：-2
```

结果：

* 玩家死亡优先；
* 本局失败；
* 不视为成功击败敌人；
* 不发放奖励。

### Example 4: 玩家存活并击败敌人

```text
最终玩家 HP：1
最终敌人 HP：0
```

结果：

* 玩家存活；
* 当前敌人被击败；
* 正常进入奖励、下一敌人或一局胜利流程。

## Scope and Non-Decisions

本文没有接受以下未来系统：

* 复活；
* 免死；
* 致命伤害保护；
* 临时超出最大 HP；
* 玩家死亡后的特殊奖励；
* 同归于尽胜利；
* 同归于尽平局；
* 失败后继续当前 run；
* 检查点或重试系统。

如果未来正式加入这些机制，应重新评估本 GDR 中相关优先级，但不应在没有正式设计决策时静默改变当前规则。

---

## Revisit Criteria

出现以下情况时，可以重新评估本决策：

* 正式加入复活或免死机制；
* 正式加入“受到致命伤害时”触发的效果；
* 正式加入同归于尽构筑；
* 正式加入特殊平局或多种终局；
* 结算后阶段的职责发生重大变化；
* round pipeline 不再采用当前固定七阶段结构；
* 玩家死亡不再直接等同于一局失败。

在这些情况出现之前，当前决策保持有效。
