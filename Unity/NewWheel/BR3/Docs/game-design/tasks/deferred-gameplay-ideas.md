# 延期玩法想法清单（Deferred Gameplay Ideas）

## 文档目的

本文用于记录当前 **未进入第一版 demo 正式实现范围**，但具有潜在设计价值的玩法想法、系统方向和待验证问题。

本文不是当前版本的正式规则文档。

当前正式生效的玩法规则以以下文档为准：

- `Docs/game-design/game-rules-locked.md`
- `Docs/game-design/trait-list.md`
- `Docs/game-design/enemy-and-reward-rules.md`

如果本文内容与正式规则文档冲突，应以正式规则文档为准。

---

## 使用原则

### 1. 本文记录的是“延期”而不是“已接受”

出现在本文中的内容，只表示它们：

- 在设计讨论中出现过
- 被认为可能有价值
- 目前未进入 v1 demo 的正式实现范围

它们不应被程序设计或程序实现默认视为当前需求。

### 2. 本文内容可以被移动、合并或删除

当某个想法未来被正式接受时，应：

- 将其写入对应的 canonical gameplay 文档
- 再从本文中移除，或把状态改为“已提升”

当某个想法未来被明确放弃时，也应更新本文状态。

### 3. 本文优先帮助“防遗忘”，不追求规则完全严密

本文允许保留：

- 尚未完全平衡的想法
- 尚未验证的系统方向
- 仅有部分规则草图的内容

但应尽量写清：

- 为什么它没有进入 v1
- 它未来可能提供什么价值
- 当前有哪些风险或未知问题

---

## 状态说明

本文中每条内容建议使用以下状态之一：

- **Deferred**：有价值，但明确延期到 v1 之后
- **Open Question**：是否应进入后续版本尚未确定
- **Rejected**：已经讨论过，但当前倾向于不采用
- **Promoted**：已进入正式规则文档，本文仅保留痕迹或待删除

当前文档以 **Deferred** 与 **Open Question** 为主。

---

# 一、延期 Trait 与卡牌效果方向

## 1. 先发（First Strike / First Play Boost）

- **状态**：Deferred
- **概要**：若这张牌作为本场 battle 的第 1 张打出，则在该轮获得显著强化，例如强度翻倍。
- **未进入 v1 的原因**：
  - 会进一步放大位置 1 的价值
  - 可能让早期出牌过于决定性
  - 在第一版中不如“强化 / 左移位 / 邻援”等 trait 更适合作为基础验证内容
- **未来价值**：
  - 强化“先手布局”思路
  - 形成高爆发 build
  - 与移位系统形成更复杂的顺序决策
- **主要风险**：
  - 可能压缩其他位置的策略空间
  - 需要额外注意数值平衡

---

## 2. 逼平 / 平局伤害（Tie Damage）

- **状态**：Deferred
- **概要**：当该牌与敌方平局时，仍然能造成少量伤害。
- **未进入 v1 的原因**：
  - 相比移位、邻援、吸血等核心 trait，玩法辨识度较低
  - 第一版更适合先验证“胜负改写”和“位置联动”
- **未来价值**：
  - 提供更稳健的牌型路线
  - 增强中低强度牌的可用性
  - 帮助构建“稳定压血”型 build
- **主要风险**：
  - 如果数值偏高，可能削弱石头剪刀布胜负关系的存在感

---

## 3. 更强的保底伤害修饰类 Trait

- **状态**：Deferred
- **概要**：例如“保底伤害 +1”“即使差值不高也能造成更多固定伤害”等。
- **未进入 v1 的原因**：
  - 相较于当前 trait，识别度较弱
  - 容易变成单纯数值叠加，不利于 v1 突出核心机制
- **未来价值**：
  - 可以服务于低强度、快节奏、持续压制类牌组
- **主要风险**：
  - 容易与基础伤害规则重叠
  - 需要小心避免变成“更好的强化”

---

## 4. 更复杂的移位 Trait

- **状态**：Deferred
- **概要**：例如：
  - 跳过一个位置进行移位
  - 同时影响两个位置
  - 进场时交换最左/最右牌
- **未进入 v1 的原因**：
  - 当前 v1 先用左移位 / 右移位验证移位玩法核心
  - 更复杂的移位会显著增加玩家推演负担和程序复杂度
- **未来价值**：
  - 可以让移位系统更有深度
  - 形成更高上限的版面构筑玩法
- **主要风险**：
  - 规则学习成本明显上升
  - 很可能需要更强的 UI 支持与日志呈现

---

## 5. 跨战斗永久削弱敌方牌库的 Trait

- **状态**：Deferred
- **概要**：例如：
  - 永久禁用敌方某一张牌
  - 永久降低敌方某类型牌的强度
  - 永久封锁敌方某位置曾使用过的牌
- **未进入 v1 的原因**：
  - 需要显式展示敌方牌库状态
  - 会显著增加跨 battle 状态追踪复杂度
  - 不利于快速做出第一版 demo
- **未来价值**：
  - 强化“针对同一敌人连续作战”的感觉
  - 提供长期策略与针对性构筑
- **主要风险**：
  - 实现成本明显高于 v1 trait
  - 如果展示不清，会让玩家无法真正感知收益

---

## 6. 直接扣血类 Trait

- **状态**：Deferred
- **概要**：不依赖当前位置胜负，直接使敌方损失固定 HP。
- **未进入 v1 的原因**：
  - 与当前“按位置比较、累积结算”的核心玩法关联较弱
  - 容易稀释石头剪刀布对抗本身的重要性
- **未来价值**：
  - 可用来做 burn / 毒伤 / 侵蚀型 build
- **主要风险**：
  - 若强度过高，会弱化位置布局的重要性

---

## 7. 提升最大血量上限的 Trait / 奖励

- **状态**：Deferred
- **概要**：可提升玩家或敌人的最大 HP 上限。
- **当前说明**：目前已经明确：
  - 吸血与回血不会超过最大血量
  - 未来可能存在提升最大血量上限的内容
  - 但 demo v1 不考虑该系统
- **未进入 v1 的原因**：
  - 当前 demo 以固定 max HP 流程为前提
  - 增加 max HP 系统会影响 reward 设计、UI 展示与平衡逻辑
- **未来价值**：
  - 可作为成长路线的一部分
  - 提供“防御型 build”与长期续航路线
- **主要风险**：
  - 会对数值节奏产生系统性影响

---

# 二、延期构筑与牌库系统方向

## 8. 战前编组 / 带牌系统

- **状态**：Deferred
- **概要**：玩家拥有一个总牌库，但每场 battle 前需要从中选出部分牌带入本场。
- **未进入 v1 的原因**：
  - 当前 v1 已采用“固定总牌库 6 张，battle 内直接从 6 张中选择”的简化方案
  - 战前编组会增加 UI、交互和状态管理成本
- **未来价值**：
  - 增加针对敌人的策略性
  - 强化构筑层与 battle 层之间的联系
- **主要风险**：
  - 对节奏有打断作用
  - 若牌库不够大，编组价值不明显

---

## 9. 更大的总牌库与抽牌系统

- **状态**：Deferred
- **概要**：允许玩家持有超过 6 张牌，并在 battle 内通过抽牌、手牌、弃牌等机制使用。
- **未进入 v1 的原因**：
  - 明显增加系统复杂度
  - 会把设计重点从“3 个位置的布局”转向“手牌资源管理”
- **未来价值**：
  - 为正式版提供更丰富的卡牌 roguelike 体验
- **主要风险**：
  - 很可能偏离当前 demo 的核心乐趣验证目标

---

## 10. 更自由的显式融合系统

- **状态**：Deferred
- **概要**：允许玩家在奖励阶段主动选择两张同类型牌进行融合，而不是把“融合结果”隐藏在升级/替换逻辑之后。
- **未进入 v1 的原因**：
  - v1 为了减少交互复杂度，采用一步完成的 Upgrade / Replace
  - 显式融合会带来二次选择目标的问题
- **未来价值**：
  - 增强构筑表达感
  - 让同类型多张牌之间的取舍更有意义
- **主要风险**：
  - 玩家操作负担上升
  - 需要更明确的牌组管理界面

---

# 三、延期敌人系统方向

## 11. 敌方 Trait 系统

- **状态**：Deferred
- **概要**：让敌方牌也具备 trait，而不仅仅是基础类型和强度。
- **未进入 v1 的原因**：
  - 会显著增加玩家计算敌方行动的认知负担
  - 当前 demo 优先让玩家专注于自己的牌与版面操作
- **未来价值**：
  - 增强敌人个性与关卡差异
  - 为后续内容扩展提供更丰富空间
- **主要风险**：
  - 可能让信息量过大
  - 需要更清晰的敌方 UI 展示方式

---

## 12. 更复杂的敌人 AI / 倾向系统

- **状态**：Deferred
- **概要**：敌人不再只是在 battle 开始时洗牌并按顺序出前 3 张，而是：
  - 根据玩家历史出牌调整倾向
  - 具有偏科牌型偏好
  - 具有特定行为模式
- **未进入 v1 的原因**：
  - v1 的重点是验证玩家侧的核心玩法
  - 复杂 AI 会增加程序实现和测试工作量
- **未来价值**：
  - 增强不同 enemy 的辨识度
  - 提高 replay value
- **主要风险**：
  - 需要更多 battle 数据与可解释性支持

---

## 13. 敌人牌库可见化与针对性削弱系统

- **状态**：Open Question
- **概要**：是否应在后续版本中更明确地向玩家展示敌方固定牌库，并允许围绕该牌库进行策略性压制。
- **当前倾向**：有潜力，但不纳入 v1。
- **主要问题**：
  - 应该展示多少敌人信息？
  - 是否会让玩法更有策略性，还是只是增加算题负担？

---

# 四、延期奖励与成长系统方向

## 14. 奖励刷新（Refresh / Reroll）

- **状态**：Deferred
- **概要**：允许玩家在一次 reward offer 中刷新选项。
- **未进入 v1 的原因**：
  - 会增加状态与 UI 分支
  - 当前 v1 更适合用“后台合法生成 + Skip”来解决糟糕选项问题
- **未来价值**：
  - 降低 reward feel-bad
  - 提高 build targeting 能力
- **主要风险**：
  - 若刷新过于便宜，会削弱随机性

---

## 15. 商店 / 金币系统

- **状态**：Deferred
- **概要**：战斗获得金币，之后在商店购买或刷新卡牌，类似更完整的 roguelike 经济系统。
- **未进入 v1 的原因**：
  - 明显增加系统复杂度
  - 当前 v1 使用 3 选 1 reward 更利于快速验证玩法
- **未来价值**：
  - 提高成长自由度
  - 让“保留钱 / 消费 / 刷新”形成新的策略层
- **主要风险**：
  - 可能让资源管理喧宾夺主

---

## 16. 多种奖励 profile / 更复杂权重系统

- **状态**：Deferred
- **概要**：不同 enemy、不同阶段、不同 build 下使用不同 reward 权重或 profile。
- **未进入 v1 的原因**：
  - 当前 v1 先使用固定 reward structure
  - 复杂 profile 暂不必要
- **未来价值**：
  - 提高成长节奏控制能力
  - 让特定敌人更像一个设计好的关卡节点
- **主要风险**：
  - 难以调试与验证

---

## 17. 最终敌人剩余奖励转化机制

- **状态**：Open Question
- **概要**：当前规则是“最终敌人被击败后剩余奖励忽略”。未来是否需要把这部分奖励转化为：
  - 通关评分
  - 结算加成
  - 展示性奖励
- **当前正式规则**：忽略。
- **未来价值**：
  - 让快速击败最终敌人的体验更完整
- **主要风险**：
  - 可能引入不必要的复杂度，削弱 demo 的简洁性

---

# 五、延期流程与模式方向

## 18. 无尽模式

- **状态**：Deferred
- **概要**：击败当前 3 个敌人后继续挑战更多敌人。
- **未进入 v1 的原因**：
  - v1 目标是短流程、可快速验证的 demo
  - 当前更重视单次 run 的闭环完整性
- **未来价值**：
  - 增加可重复游玩性
  - 便于测试成长上限和极端 build
- **主要风险**：
  - 需要更多敌人内容与成长平衡支持

---

## 19. 更多敌人数 / 更多战斗阶段

- **状态**：Deferred
- **概要**：未来可增加敌人数，或提高每个敌人的 battle 次数。
- **未进入 v1 的原因**：
  - 当前 3 个敌人 × 最多 3 场 battle 的结构已足够支撑核心验证
- **未来价值**：
  - 增加 run 长度与 build 成型空间
- **主要风险**：
  - 若核心玩法未足够扎实，内容延长只会放大重复感

---

# 六、当前明确不建议直接推进到 v1 的方向

## 20. 让敌人复杂度与玩家复杂度同时上升

- **状态**：Rejected（针对 v1）
- **概要**：同时增加玩家 trait 深度、敌方 trait 深度、敌方 AI 深度。
- **原因**：
  - 会显著增加第一版 demo 的理解与实现负担
  - 不符合“快速完成最小可玩 demo”的目标

---

## 21. 在 v1 中引入大牌库 + 抽牌 + 商店 + 多层奖励分支

- **状态**：Rejected（针对 v1）
- **概要**：把更完整的卡牌 roguelike 结构一次性加入 v1。
- **原因**：
  - 会分散对核心机制的验证
  - 让程序实现成本大幅上升

---

# 七、建议的后续维护方式

## 当某个想法被正式接受时

应执行：

1. 把它写入对应 canonical gameplay 文档
2. 更新相关 engineering 文档（如有必要）
3. 将本文对应条目的状态改为 `Promoted`，或直接删除该条目

## 当某个想法被明确放弃时

应执行：

1. 将其状态改为 `Rejected`
2. 简短记录放弃原因，避免未来重复讨论

## 当某个想法仍然模糊时

可以先保留在本文，但建议补充：

- 需要验证什么
- 哪些实现或平衡问题尚未想清楚
