# 全局修正头脑风暴规范

## 文档目的

本文档用于指导 BR3 后续关于 **修正 / Modifier**、**全局修正 / Global Modifier**、**物品词条 / Item Modifier**、**角色被动 / Character Passive** 等内容的玩法设计讨论。

本文不是当前版本的正式玩法规则文档，也不是实现规格文档。它的主要作用是：

- 统一 Chat D 头脑风暴时描述修正的方式；
- 帮助 Chat A 后续筛选、评估和平衡候选修正；
- 降低不同流派讨论之间的重复；
- 避免设计出违反当前轮内结算阶段顺序的修正；
- 为未来物品、角色、敌人规则、模式规则等系统提供一套可复用的讨论方法。

本文档应与 `Docs/game-design/glossary.md` 配合使用。术语定义以术语表为准，本文只说明如何在修正设计讨论中使用这些术语。

---

## 适用范围

本文适用于以下内容的头脑风暴和评估：

- 物品词条；
- 角色被动；
- 敌人规则；
- 模式规则；
- 其他全局修正；
- 未来可能出现的牌修正、槽位修正等候选设计。

本文尤其适合 Chat D 使用。Chat D 可以更开放地探索长期玩法可能性，但仍应使用本文档中的基本规范来避免讨论结果过于散乱。

Chat A 可以使用本文档作为评估标准，对 Chat D 产出的修正进行筛选、平衡、简化或重写。

---

## 与术语表的关系

`Docs/game-design/glossary.md` 是术语定义文档。

本文档不重复维护完整术语表。若本文中的术语与术语表冲突，应以术语表为准。

本文档中的英文术语主要用于帮助未来代码命名和设计沟通。除非有明确原因，未来代码命名应尽量遵循术语表中的英文。

---

## Chat D 与 Chat A 的分工

### Chat D

Chat D 负责：

- 打开设计空间；
- 尽量发现有趣的流派方向；
- 提出物品、角色、敌人规则、模式规则等候选修正；
- 记录修正之间可能存在的混合触发与涌现联动；
- 粗略判断某些修正是否明显过于复杂或违反当前核心规则。

Chat D 不需要：

- 完成最终数值平衡；
- 决定某个修正是否进入下一个实现版本；
- 为每个头脑风暴想法都写完整规格；
- 强行把所有想法都转化为可实现任务。

### Chat A

Chat A 负责：

- 判断哪些修正适合进入近期版本；
- 对候选修正进行更严格的平衡性评估；
- 删除、合并、简化或重写过于复杂的设计；
- 决定哪些内容应写入 canonical gameplay 文档；
- 决定哪些内容应保留在延期设计文档中。

# 1. 修正设计轴

本节记录设计修正时可反复使用的通用分析维度。

这些设计轴不是某一个流派专属的方法。它们可以用于平局流、移位流、低血流、反伤流、毒伤流、奖励经济流等后续所有流派讨论。

Chat D 在头脑风暴时，可以用这些设计轴打开设计空间。Chat A 在评估时，可以用这些设计轴检查候选修正是否存在重复、歧义、时序风险、UI 风险或平衡风险。

---

## 1.1 玩家体验轴

玩家体验轴用于回答：

> 这个修正主要让玩家体验到什么？

常见方向包括：

- 输出：把某个行为转化为伤害或击败效率；
- 续航：提供治疗、减伤、护盾、护甲、承伤缓冲；
- 蓄力：把当前行为积累为之后释放的收益；
- 成长：让牌、角色或构筑在更长跨度上变强；
- 控制：削弱敌人、限制敌人、改变敌人后续威胁；
- 阵型：强调槽位、相邻、对位、多槽结构；
- 奖励经济：影响奖励生成、奖励质量、奖励选择或构筑收束；
- 规则改写：改变基础规则本身的含义。

规则改写型通常风险较高，应特别谨慎。它适合少量核心被动、高稀有修正、敌人规则或模式规则，不应轻易作为普通物品词条大量出现。

---

## 1.2 触发组合轴

触发组合轴用于回答：

> 这个修正是只属于一个流派，还是同时要求多个流派条件？

常见类型包括：

- 纯流派修正：只要求一个核心条件；
- 混合触发修正：触发条件同时包含两个或更多流派概念；
- 涌现联动：多个独立修正自然组合后产生额外效果。

示例：

```text
纯流派修正：
发生槽位平局时，敌人受到 1 点保底伤害。

混合触发修正：
槽位 1 发生槽位平局时，敌人受到 2 点保底伤害。

涌现联动：
一个修正让槽位平局造成保底伤害，另一个修正让保底伤害 +1。
```

设计讨论时，应系统展开混合触发修正，但不需要穷尽所有涌现联动。涌现联动只记录特别有趣或特别危险的例子。

---

## 1.3 来源轴

来源轴用于回答：

> 这个效果归属于谁？

当前最重要的是：

- 伤害源；
- 治疗源。

候选伤害源包括：

- 牌；
- 槽位；
- 物品；
- 角色；
- 敌人；
- 系统。

候选治疗源包括：

- 牌；
- 物品；
- 角色；
- 敌人；
- 系统。

默认原则：

- 牌源伤害可以触发该牌自身的伤害相关特质或效果；
- 非牌源伤害默认不触发牌特质，除非修正特别说明；
- 牌源治疗可以触发该牌自身的治疗相关效果；
- 非牌源治疗默认不触发牌特质，除非修正特别说明。

未来如果需要更细的统计或联动，可以继续讨论强化源、削弱源、控制源等概念。但在当前阶段，伤害源和治疗源优先级最高。

---

## 1.4 伤害类别轴

伤害类别轴用于回答：

> 这段伤害属于哪一类规则？

当前建议伤害类别包括：

- 槽位战斗伤害；
- 保底伤害；
- 反伤；
- 毒伤；
- 治疗转伤；
- 固定伤害；
- 系统伤害。

这些类别目前是设计讨论中的建议列表，不是已锁定的实现枚举。

设计原则：

> 不要为每个触发条件创建新的伤害类别。

例如，通常不应创建：

- 平局固定伤害；
- 移位固定伤害；
- 治疗固定伤害；
- 槽位 1 固定伤害。

更推荐将“为什么触发”和“伤害类别”分开：

```text
触发条件：发生槽位平局
伤害类别：保底伤害
```

```text
触发条件：发生移位
伤害类别：固定伤害
```

触发条件说明为什么发生，伤害类别说明这段伤害在规则系统中属于哪类。

---

## 1.5 作用时机轴

作用时机轴用于回答：

> 这个修正在什么时候结算？

常见作用时机包括：

- 指定轮内结算阶段开始时；
- 指定轮内结算阶段结束时；
- 槽位结果产生后；
- 合并伤害阶段后；
- 结算后阶段；
- 本轮结束时；
- 下一轮开始时；
- 战斗结束时；
- 击败敌人时；
- 生成奖励时；
- 选择奖励时。

作用时机必须尊重当前轮内结算阶段顺序。

尤其要避免：

> 在槽位结果已经产生后，回头修改本轮已经参与槽位结算的强度、胜负或槽位结果。

如果某个修正在槽位结果产生后触发，通常应改为：

- 影响下一轮；
- 造成额外伤害；
- 产生治疗；
- 记录状态；
- 影响后续奖励；
- 影响后续战斗。

---

## 1.6 持续时间与释放跨度轴

持续时间与释放跨度轴用于回答：

> 这个修正的效果持续到什么时候？它在多长的时间跨度后释放收益？

常见持续时间包括：

| 持续时间 | 说明 | 设计倾向 |
|---|---|---|
| 本次触发 | 只在当前触发事件中生效 | 最简单，风险低 |
| 本轮 | 当前轮内生效 | 需要注意轮内阶段顺序 |
| 下一轮 | 下一轮生效或释放 | 很适合短期蓄力 |
| 固定轮次 | 例如第 3 轮释放 | 适合利用 BR3 的三轮结构 |
| 本场战斗 | 本场战斗内保留或累计 | 构筑感较强，需要展示 |
| 当前敌人阶段 / 跨战斗 | 同一个敌人的多场战斗之间保留 | 长线经营感强，数值应更高，数量不宜多 |
| 一局游戏 / 跨敌人 | 可以带到后续敌人 | 接近 run 内永久收益，风险较高 |
| 跨局游戏 | 下一局游戏仍然保留 | 属于局外成长，早期版本不考虑 |

一般原则：

- 持续时间越长，通常越需要更高收益来补偿延迟满足；
- 持续时间越长，越需要 UI 或日志清楚展示；
- 持续时间越长，越容易产生拖延战斗、长期数值失控或记忆负担；
- 跨战斗和跨敌人修正应少量使用，并交给 Chat A 重点评估。

---

## 1.7 释放条件轴

释放条件轴用于回答：

> 已经积累的收益在什么条件下兑现？

常见释放条件包括：

- 自动释放；
- 下一轮释放；
- 第 N 轮释放；
- 本轮结束释放；
- 战斗结束释放；
- 槽位胜利释放；
- 槽位失败释放；
- 受到伤害释放；
- 造成治疗释放；
- 低血时释放；
- 生成奖励时释放；
- 选择某类奖励时释放。

设计时应特别注意：

1. 若释放条件依赖多个槽位同时结算，例如“下一次槽位胜利”，需要明确槽位顺序或改成无顺序歧义的写法。
2. 若释放条件过于隐蔽，需要额外 UI / 日志支持。
3. 若释放条件过于稳定，可能变成无脑收益。
4. 若释放条件过难，玩家可能感受不到修正价值。

示例：

```text
平局积累，槽位胜利释放：
每发生 1 次槽位平局，获得 1 层均势。
下一次槽位胜利时，消耗均势并造成额外伤害。
```

这种设计可以避免玩家完全放弃槽位胜利，让平局成为准备动作，槽位胜利成为释放动作。

---

## 1.8 可见性与 UI 成本轴

可见性与 UI 成本轴用于回答：

> 玩家是否能看懂这个修正已经触发、正在持续、即将释放或已经释放？

常见可见性需求包括：

- 只需要战斗日志；
- 需要状态图标；
- 需要状态图标上的层数数字；
- 需要版面标记；
- 需要槽位标记；
- 需要下一轮预览；
- 需要奖励生成解释；
- 需要战斗结束汇总。

设计原则：

- 即时伤害、即时治疗等效果，通常只需要日志和数值反馈；
- 下一轮生效的状态，至少需要日志，正式 UI 中最好有状态提示；
- 层数型资源需要能显示当前层数；
- 跨战斗或跨敌人状态需要更明确的持久状态展示；
- 奖励经济类修正如果改变奖励生成，应尽量解释“为什么这次奖励变了”。

Chat D 可以先记录 UI 需求和风险，不需要完成 UI 方案。具体 UI 设计可交由 Chat A 或后续专门 UI 讨论。

---

## 1.9 数值与限制轴

数值与限制轴用于回答：

> 这个修正如何防止过强、过弱或无限滚雪球？

常见限制方式包括：

- 无额外限制；
- 每轮最多 1 次；
- 每场战斗最多 1 次；
- 每张牌每场战斗最多 1 次；
- 同名修正不叠加；
- 层数上限；
- 只在第 2 / 第 3 轮生效；
- 只在敌人未被击败时生效；
- 只在玩家低血时生效；
- 只影响下一次触发；
- 战斗结束清空；
- 当前敌人阶段结束清空。

如果没有额外限制，应写“无”，而不是重复写规则本身已经隐含的限制。

例如某个槽位每轮本来只会产生一次槽位结果，则不必写“每个槽位每轮最多触发 1 次”，除非该限制用于处理同名修正叠加或其他特殊情况。

---

## 1.10 设计轴使用建议

Chat D 头脑风暴时，不需要对每个修正都完整遍历所有设计轴。

建议用法：

1. 先用玩家体验轴确定这个修正想带来的体验；
2. 再用触发组合轴判断它是纯流派修正、混合触发修正，还是涌现联动；
3. 如果涉及伤害或治疗，再检查来源轴与伤害类别轴；
4. 如果效果延后释放，再检查持续时间与释放条件；
5. 如果状态需要记忆或累计，再检查可见性与 UI 成本；
6. 如果可能过强或重复触发，再检查数值与限制轴；
7. 最后检查是否违反轮内结算阶段顺序。

这些设计轴的作用是帮助发现设计空间和风险，不是要求每条修正都写成完整规格。

# 2. 修正描述模板

修正可以使用两种模板描述：

- **简写模板**：用于 Chat D 快速头脑风暴；
- **完整模板**：用于候选修正准备移交 Chat A 或进入较严肃评估时。

## 1.1 简写模板

Chat D 大量发散时，优先使用简写模板。

```markdown
### 修正名

**效果概述：**
一句话描述这个修正，尽量包含触发条件、作用对象和主要效果。

- 流派：
- 可用载体：
- 关键规则：
- 设计意图：
- 联动备注：
- 风险：
```

### 适用情况

简写模板适合：

- 初次提出想法；
- 同一流派下快速列举多个方向；
- 尚不确定是否值得深入的修正；
- 不涉及复杂伤害源、治疗源、伤害类别或作用时机的修正。

### 示例

```markdown
### 平衡之刃

**效果概述：**
发生槽位平局时，敌人受到 1 点保底伤害。

- 流派：平局流
- 可用载体：物品词条 / 角色被动
- 关键规则：槽位平局触发；伤害类别为保底伤害。
- 设计意图：让平局成为稳定低风险输出来源。
- 联动备注：可与“保底伤害 +1”类修正联动。
- 风险：若保底伤害容易叠高，平局输出可能过强。
```

---

## 1.2 完整模板

完整模板用于更严谨地描述候选修正。准备交给 Chat A 评估的重点修正，建议尽量补全完整模板。

```markdown
### 修正名

**效果概述：**
一句话描述这个修正，尽量包含触发条件、对象、主要效果和关键类别。

- 流派：
- 可用载体：
- 触发条件：
- 作用时机：
- 作用对象类型：
- 作用范围：
- 筛选条件：
- 效果细则：
- 伤害源：
- 伤害类别：
- 治疗源：
- 持续时间：
- 限制：
- 设计意图：
- 联动备注：
- 相近修正区别：
- 风险：
```

并非每个字段都必须存在。若某个字段与该修正无关，可以写“无”，也可以在简写阶段省略。

---

# 3. 完整模板字段说明

## 2.1 效果概述

一句话快速说明修正做什么。

效果概述应尽量包含：

- 触发条件；
- 主要作用对象；
- 主要效果；
- 若涉及伤害，尽量说明伤害类别。

效果概述用于快速查阅，不承担所有规则严谨性。更细节的歧义应在后续字段中消除。

### 示例

```markdown
**效果概述：**
发生槽位平局时，敌人受到 1 点保底伤害。
```

```markdown
**效果概述：**
拥有吸血的牌发生槽位平局时，该牌对敌人造成 1 点保底伤害。
```

---

## 2.2 流派

说明该修正服务于哪些流派。

示例：

- 平局流；
- 移位流；
- 平局流 / 位置流；
- 低血流 / 续航流；
- 奖励经济流。

如果一个修正的触发条件同时包含两个流派概念，应标记为混合触发修正。

---

## 2.3 可用载体

说明这个修正可能以什么形式出现。

常见载体包括：

- 物品词条；
- 角色被动；
- 敌人规则；
- 模式规则；
- 未来可能出现的牌修正或槽位修正。

同一个修正原型可以有多个潜在载体。Chat D 中不必过早锁定载体。

---

## 2.4 触发条件

说明什么时候检查或触发该修正。

示例：

- 发生槽位平局时；
- 槽位 1 发生槽位平局时；
- 一张牌发生移位后；
- 本轮至少发生 2 次槽位平局时；
- 玩家 HP 低于一半时；
- 战斗结束时；
- 生成下一次奖励时。

触发条件不应混入伤害源、治疗源或作用对象类型。

---

## 2.5 作用时机

说明修正在哪个时间点结算。

常见作用时机包括：

- 指定轮内结算阶段开始时；
- 指定轮内结算阶段结束时；
- 槽位结果产生后；
- 合并伤害阶段后；

## 2.11 伤害类别

当修正造成伤害时，应明确伤害类别。

当前建议伤害类别包括：

- 槽位战斗伤害；
- 保底伤害；
- 反伤；
- 毒伤；
- 治疗转伤；
- 固定伤害；
- 系统伤害。

这些类别目前是设计讨论中的建议列表，不是已锁定的实现枚举。

### 设计原则

不要为每个触发条件创建新的伤害类别。

例如，通常不应创建：

- 平局固定伤害；
- 移位固定伤害；
- 治疗固定伤害；
- 槽位 1 固定伤害。

更推荐写成：

```markdown

伤害类别：保底伤害
```

或：

```markdown
触发条件：发生移位
伤害类别：固定伤害
```

触发条件说明为什么发生，伤害类别说明这段伤害属于哪类规则。

## 2.13 持续时间

持续时间说明效果持续多久。

- 本次触发；
- 本轮；

- 本场战斗；
- 当前敌人阶段；
- 一局游戏；
- 永久。

## 2.14 限制

限制用于说明额外触发次数、叠加、频率或边界。

示例：

- 无；
- 每轮最多 1 次；
- 每场战斗最多 1 次；
- 每张牌每场战斗最多 1 次；
- 同名修正不叠加；
- 即使拥有多个同名修正，每轮也只触发 1 次。

如果没有额外限制，应写“无”，而不是写规则本身已经隐含的限制。

例如某个槽位每轮本来只会产生一次槽位结果，则不必写“每个槽位每轮最多触发 1 次”，除非该限制是为了处理同名修正叠加或其他特殊情况。

---

## 2.15 设计意图

设计意图说明这个修正希望制造什么玩家行为或体验。

示例：

- 让槽位平局成为稳定低风险输出来源；
- 鼓励玩家围绕移位制造后续收益；
- 让低血状态成为可主动利用的风险资源；
- 让奖励选择更容易朝某个流派收束。

设计意图比数值更重要。Chat A 后续可以调整数值，但应尽量保留明确的体验目标。

---

## 2.16 联动备注

联动备注用于记录修正可能和其他规则产生的自然联动或设计联动。

示例：

- 与“保底伤害 +1”联动；
- 若伤害源是牌，则可能触发吸血；
- 与溢出治疗转伤联动后可能过强；
- 可与移位流联动，玩家通过移位主动制造槽位平局。

联动备注不需要穷尽所有组合，只记录特别有价值或特别危险的联动。

---

## 2.17 相近修正区别

相近修正区别用于说明这个修正和容易混淆的其他修正有什么差异。

此字段有助于避免文档中出现大量本质重复的修正。

示例：

```markdown
- 相近修正区别：与“平局造成 1 点固定伤害”的区别是，本修正造成 1 点保底伤害，因此会受到保底伤害强化影响。
```

```markdown
- 相近修正区别：与“均衡护符”的区别是，本修正在战斗结束后按平局次数治疗；均衡护符是在槽位平局发生后立即治疗。
```

如果没有容易混淆的相近修正，可以写“无”。

---

## 2.18 风险

风险用于记录平衡性、可读性、实现复杂度和 UI 展示问题。

示例：

- 可能削弱石头 / 剪刀 / 布胜负关系；
- 可能让战斗过慢；
- 需要较清晰的战斗日志支持；
- 可能与同名叠加后失控；
- 可能违反轮内结算阶段顺序；
- 可能需要额外 UI 展示目标或计数。

# 4. 伤害源、治疗源与伤害类别规则

## 4.1 伤害源必须明确

凡是造成伤害的修正，都应明确伤害源。

伤害源影响：

- 是否触发牌的吸血或其他伤害相关特质；
- 战斗日志如何记录；
- 后续统计中伤害归属于谁；
- 某些修正是否能识别这段伤害。

默认规则：

- 牌源伤害可以触发该牌自身的伤害相关特质；
- 非牌源伤害不触发牌特质，除非修正明确说明。

---

## 4.2 治疗源必须明确

凡是产生治疗的修正，都应明确治疗源。

治疗源影响：

- 是否触发未来可能存在的治疗相关效果；
- 战斗日志如何记录；
- 后续统计中治疗归属于谁。

默认规则：

- 牌源治疗可以触发该牌自身的治疗相关效果；
- 非牌源治疗不触发牌特质，除非修正明确说明。

---

## 4.3 伤害类别必须明确

凡是造成伤害的修正，都应明确伤害类别。

伤害类别影响：

- 哪些伤害强化修正可以影响它；
- 它是否属于反伤、毒伤、治疗转伤等特定构筑路线；
- 后续统计和日志如何分类。

设计时应避免为每个触发来源创建独立伤害类别。

例如，不推荐把以下内容都做成独立伤害类别：

- 平局伤害；
- 移位伤害；
- 位置 1 伤害；
- 治疗伤害。

更推荐用：

- 触发条件说明为什么发生；
- 伤害类别说明这段伤害在规则上属于哪类。

---

## 4.4 保底伤害与固定伤害

**保底伤害** 是一种伤害类别。

例如：

```text
造成 1 点保底伤害。
```

这里的 `1` 是基础数值。如果玩家拥有“保底伤害 +1”类修正，实际伤害可以被提高。

**固定伤害** 是不归入其他伤害类别的固定数值伤害。它通常不被保底伤害、反伤、毒伤等类别强化影响。

例如：

```text
战斗开始时，敌人受到 2 点固定伤害。
```

如果设计者希望某段伤害参与某个伤害类别的联动，就不应使用固定伤害，而应明确使用对应类别。

---

# 6. 轮内结算阶段顺序约束

当前一轮结算遵循固定阶段顺序：

1. 入场阶段；
2. 固定自属性阶段；
3. 移位阶段；
4. 版面派生阶段；
5. 槽位结算阶段；
6. 合并伤害阶段；
7. 结算后阶段。

修正设计必须尊重这个顺序。

## 5.1 禁止回头修改已结算强度

如果某个修正在槽位结果产生后触发，它不应回头修改本轮已经参与槽位结算的强度、胜负或槽位结果。

不推荐设计：

```text
槽位 1 发生槽位平局后，槽位 2 和槽位 3 的玩家牌本轮 +1 强度。
```

原因是：

- 槽位平局要到槽位结算阶段才知道；
- 本轮强度已经在版面派生阶段计算完毕；
- 如果此时再修改强度，就需要重新计算其他槽位结果；
- 这会让规则、程序和玩家理解都变混乱。

## 5.2 推荐改写方式

如果某个设计看起来会违反阶段顺序，可以改成：

- 影响下一轮；
- 影响本轮结算后收益；
- 造成额外伤害；
- 产生治疗；
- 改变下一次奖励；
- 改变后续战斗；
- 记录本轮事件，并在未来阶段消费。

推荐改写：

```text
槽位 1 发生槽位平局后，下一轮新打出的玩家牌 +1 强度。
```

```text
槽位 1 发生槽位平局后，本轮结束时玩家回复 1 HP。
```

```text
槽位 1 发生槽位平局后，敌人受到 1 点保底伤害。
```

# 6. 同名修正叠加规则

当前讨论默认：

> 同名修正可以叠加，除非该修正明确说明不叠加或有触发上限。

例如：

```text
两个均衡护符可以分别触发。
```

如果某个修正不允许叠加，应在“限制”字段中明确写出。

示例：

```markdown
- 限制：同名修正不叠加。
```

或者：

```markdown
- 限制：每轮最多触发 1 次，即使拥有多个同名修正。
```

强力修正、倍增型修正、容易导致循环触发的修正，应优先考虑添加叠加限制或触发上限。

---

# 7. 重复修正整理原则

头脑风暴中应避免把本质相同的修正拆成大量重复条目。

## 7.1 合并相近变体

如果两个修正只有轻微数值或结算类别差异，可以合并成一个修正条目，并在其中列出子方案。

例如：

```markdown
### 平局伤害

**效果概述：**
发生槽位平局时，敌人受到少量伤害。

可选版本：
- 版本 A：造成 1 点保底伤害，可与保底伤害强化联动。
- 版本 B：造成 1 点固定伤害，不参与保底伤害强化。
```

## 7.2 需要分开时说明区别

如果两个修正确实需要分开列举，应在“相近修正区别”字段中说明它们的设计差异。

例如：

- 即时治疗 vs 战斗结束治疗；
- 槽位源伤害 vs 牌源伤害；
- 保底伤害 vs 固定伤害；
- 影响下一轮强度 vs 造成本轮结算后伤害。

---

# 8. 有序流派组合讨论法

为了避免流派组合讨论无限重复，Chat D 后续讨论流派时应使用 **有序流派组合讨论法**。

## 8.1 基本规则

先确定一个流派讨论顺序。

示例：

1. 平局流；
2. 移位流；
3. 位置流；
4. 邻接阵型流；
5. 单类型偏科流；
6. 三色均衡流；
7. 特质专精流；
8. 低血 / 续航 / 风险流；
9. 压制 / 控制流；
10. 爆发 / 终结流；
11. 奖励 / 经济 / 构筑操控流；
12. 敌人针对流；
13. 随机 / 混乱流。

讨论某个流派时，应讨论它与后续尚未深入讨论流派之间的混合触发修正。

后续讨论其他流派时，不再重复讨论它与已经完成流派之间的组合。

---

## 8.2 示例

讨论平局流时，可以讨论：

- 平局 + 移位；
- 平局 + 位置；
- 平局 + 邻接；
- 平局 + 单类型；
- 平局 + 三色均衡；
- 平局 + 特质专精；
- 平局 + 低血；
- 平局 + 压制；
- 平局 + 爆发；
- 平局 + 奖励经济；
- 平局 + 敌人针对。

之后讨论移位流时，不再重复“移位 + 平局”，只讨论：

- 移位 + 位置；
- 移位 + 邻接；
- 移位 + 单类型；
- 移位 + 三色均衡；
- 等等。

这样可以系统覆盖组合空间，同时避免重复讨论。

---

# 9. 混合触发与涌现联动

## 9.1 混合触发修正

混合触发修正是设计者主动设计出的组合修正。

例如：

```text
槽位 1 发生槽位平局时，敌人受到 1 点保底伤害。
```

它的触发条件同时包含：

- 槽位平局；
- 槽位 1。

混合触发修正需要在流派文档中系统展开。

---

## 9.2 涌现联动

涌现联动是多个独立修正自然组合后产生的联动。

例如：

```text
修正 A：发生槽位平局时，敌人受到 1 点保底伤害。
修正 B：保底伤害 +1。
```

这两个修正同时存在时，平局造成的保底伤害会被提高。

涌现联动不需要穷尽列举。只需要记录：

- 特别有趣的组合；
- 特别危险的组合；
- 可能影响平衡或实现的组合；
- 可能让 Chat A 需要额外注意的组合。

# 10. 进入 Chat A 评估前的建议检查清单

当一个修正准备从 Chat D 移交给 Chat A 时，建议至少检查以下问题。

## 10.1 规则清晰性

- 效果概述是否能让人快速理解？
- 触发条件是否明确？
- 作用时机是否明确？
- 作用对象类型是否明确？
- 作用范围和筛选条件是否足够清楚？

## 10.2 战斗系统兼容性

- 是否违反当前轮内结算阶段顺序？
- 是否要求回头修改本轮已经结算过的强度或槽位结果？
- 是否需要额外事件记录或计数器？
- 是否需要新的 UI 展示？

## 10.3 伤害与治疗严谨性

如果修正造成伤害：

- 是否写明伤害源？
- 是否写明伤害类别？
- 是否会触发牌特质？
- 是否会参与某类伤害强化？

如果修正产生治疗：

- 是否写明治疗源？
- 是否受最大 HP 限制？
- 是否可能触发治疗相关联动？

## 10.4 流派价值

- 它是否改变玩家评价局面的方式？
- 它是否提供了新的构筑路线？
- 它是否只是单纯数值增强？
- 它是否与已有流派有清晰区分？

## 10.5 平衡与复杂度风险

- 是否可能过强？
- 是否可能拖慢战斗？
- 是否需要复杂 UI 才能理解？
- 是否和同名叠加后失控？
- 是否和某些涌现联动过强？

# 11. 文档维护建议

## 11.1 本文档的状态

本文档是玩法设计讨论规范，不是正式玩法规则。

如果 Chat A 接受本文档的流程规范，可将其作为长期设计方法文档保留。

## 11.2 与具体流派文档的关系

具体流派文档，例如平局流文档，应使用本文档中的术语和模板，但不需要重复解释模板本身。

具体流派文档应重点记录：

- 流派设计意图；
- 子方向；
- 修正候选；
- 角色候选；
- 混合触发修正；
- 少量涌现联动；
- 风险；
- 早期版本候选。

## 11.3 与 canonical gameplay 文档的关系

本文中的修正、流派和伤害类别示例不代表已经进入正式规则。

当某个修正被 Chat A 接受并准备实现时，应写入对应的 canonical gameplay 文档或新的设计决策文档。

## 11.4 与术语表的关系

若本文档中出现新术语，应同步考虑是否加入 `Docs/game-design/glossary.md`。

术语表负责定义术语，本文档负责说明讨论方法。
