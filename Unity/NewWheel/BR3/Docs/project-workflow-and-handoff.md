# BR3 Project Workflow and Handoff

## 1. 文档目的与适用范围 `[Shared]`

本文档用于记录 BR3 项目中，用户与多个 ChatGPT 会话之间已经形成的协作方式、职责边界、交接规则、恢复流程和文件管理规范。

本文档的目标不是描述游戏本身，而是确保：

- 不同会话知道自己负责什么；
- 用户与各会话之间的工作方式能够复用；
- 一个会话失效后，可以依靠文档恢复其职责与当前工作状态；
- 多个会话可以围绕独立工作范围并行推进；
- 设计、工程和实现之间的交接保持清晰；
- 聊天记录不是唯一的工作流知识来源。

本文档不是：

- canonical gameplay 规则文档；
- canonical engineering 设计文档；
- 实现规格；
- 临时聊天记录的完整归档；
- Project Instructions 的替代品。

如果本文件引用了玩法、工程或实现规则，应优先引用对应 source-of-truth 文档，而不是在本文件中复制完整规则。

## 2. 术语与命名 `[Shared]`

### 2.1 Chat 角色

BR3 项目当前使用以下会话角色：

- **Chat A**：Gameplay Design
- **Chat B**：Engineering Design
- **Chat C**：Implementation Coordination
- **Chat D**：Gameplay Brainstorming

各角色的详细职责由各自专属工作协议定义。

### 2.2 主协调会话

**主协调会话** 指某个 Chat 角色中负责总体规划、任务拆分、交接、状态维护和结果复查的主要会话。

正式称呼示例：

```text
Chat D 主协调会话
Chat B 主协调会话
```

常用简称：

```text
D 主会话
B 主会话
```

极简简称：

```text
D主
B主
```

### 2.3 专项会话

**专项会话** 指负责一个边界明确、可以独立推进的工作范围的会话。

专项范围可以是：

- 一个玩法章节；
- 一个流派主题；
- 一个平衡专题；
- 一个程序模块；
- 一份工程文档；
- 一个实现批次；
- 一组 Codex 任务；
- 一个问题调查；
- 其他适合独立处理的工作单元。

正式称呼示例：

```text
Chat D 专项会话
Chat B 专项会话
```

常用简称：

```text
D 专项会话
B 专项会话
```

极简简称：

```text
D专项
B专项
```

具体实例可以称为：

```text
第 7 章专项
奖励模块专项
Batch 3 专项
```

### 2.4 不使用“子 Chat”或“子 agent”

专项会话是用户手动创建的独立聊天，不是主协调会话自动创建、调度和回收的技术意义上的 subagent。

因此，不使用以下称呼：

```text
Chat D 子
D子
子 agent
subagent
```

以避免与真正的 agent delegation 混淆。

## 3. Source of Truth 与冲突处理 `[Shared]`

### 3.1 基本原则

聊天记录可以提供上下文，但不是稳定的 source of truth。

稳定知识应优先写入：

- canonical gameplay 文档；
- canonical engineering 文档；
- ADR / GDR；
- 全局规范文档；
- 当前流派累计文档；
- 本工作流与 handoff 文档；
- 明确的当前状态快照。

### 3.2 Gameplay 层级

Gameplay 内容建议按以下优先级理解：

```text
locked / canonical gameplay documents
> Chat A 已接受并写入正式文档的设计
> Chat D 候选设计文档
> 临时 handoff
> 单次聊天中的未落盘讨论
```

Chat D 中出现的候选不自动进入正式实现范围。

### 3.3 Engineering 层级

Engineering 内容建议按以下优先级理解：

```text
canonical engineering documents / ADR
> Chat B 已确认的工程 handoff
> Chat C implementation plan
> Codex 临时解释或建议
```

Chat C 和 Codex 不应静默修改已接受的玩法或工程设计。

### 3.4 冲突处理

发现冲突时：

1. 不静默选择自己偏好的版本；
2. 确认冲突涉及 gameplay、engineering 还是 implementation；
3. 回到对应负责角色处理；
4. 更新 source-of-truth 文档；
5. 再继续后续工作。

典型归属：

```text
玩法取舍或规则冲突 → Chat A
长期候选空间与流派讨论 → Chat D
架构、领域模型、流程、测试设计 → Chat B
实现批次、Codex 提示词、落地顺序 → Chat C
```

## 4. 通用多会话协作原则 `[Shared]`

### 4.1 主协调会话与专项会话分工

主协调会话负责：

- 规划工作范围；
- 判断哪些任务适合拆分；
- 冻结本批次共同输入；
- 生成专项任务包；
- 维护跨专项边界；
- 接收完成报告；
- 统一检查重复、冲突和遗漏；
- 规划下一批工作。

专项会话负责：

- 只处理被分配的范围；
- 按对应角色的工作协议推进；
- 不自行扩大职责；
- 不修改其他专项内容；
- 完成后返回结构化报告。

### 4.2 固定协议与动态任务包分离

固定工作方式保存在本文档。

每次创建专项会话时，主协调会话只需要生成一份动态任务包，填写当前任务的变量。

动态任务包至少应包含：

- 角色；
- 负责范围；
- 当前阶段；
- 共同基线；
- 必读文档；
- 已确认边界；
- 相关前置工作；
- 输出范围；
- 目标文件名；
- 完成后回传要求。

不建议只写：

```text
参见 handoff 文档第 X 节，负责某任务。
```

因为这不足以说明当前基线、任务阶段和专项边界。

### 4.3 冻结共同基线

同一批并行专项开始前，主协调会话应明确一个共同内容基线。

共同基线应主要由以下信息确定：

```text
文档标识
+ 文档内部内容版本
```

Repository 文件名用于说明文档在仓库中的稳定位置；ChatGPT Project 显示名只用于在 Project Sources 中定位当前上传文件。

例如：

```text
共同基线：

文档标识：
tie-build-brainstorm

Repository 文件：
tie-build-brainstorm.md

内容版本：
v06

当前 Project 显示名：
tie-build-brainstorm(6).md
```

开始专项前，主协调会话应确认：

- Project 中读取到的文档标识正确；
- 文档内部内容版本与任务包一致；
- 当前 Project 显示名指向该内容版本；
- ChatGPT 自动添加的 `(N)` 后缀没有被误认为内容版本。

在该批专项完成前：

- 不改变共同基线的设计内容；
- 不让不同专项分别使用不同内容版本；
- 不在中途把某个专项结果直接合并后重新上传给其他专项；
- 不因 Project 显示名变化而静默切换基线；
- 等整批完成后再集中合并，并在需要时更新内部内容版本。
### 4.4 专项不直接修改共同累计文档

专项会话应只输出自己的独立工作成果。

例如：

```text
tie-build-chapter-07-low-hp-sustain.md
```

而不是重新生成：

```text
tie-build-brainstorm.md
```

共同累计文档由用户在本地集中维护和合并。

### 4.5 聊天记录不是唯一依赖

专项会话应主要依赖：

- source-of-truth 文档；
- 当前共同基线；
- 本工作流文档；
- 主协调会话生成的任务包。

不应依赖：

```text
请自己在 Project 中找出主会话之前说过什么。
```

项目内聊天召回可以作为辅助，但不能代替明确的文档和 handoff。

## 5. 文件与版本管理 `[Shared]`

### 5.1 Repository 中使用稳定文件名

Repository 中的累计设计文档应优先使用稳定文件名：

```text
<archetype>-brainstorm.md
```

例如：

```text
tie-build-brainstorm.md
movement-build-brainstorm.md
```

这样可以避免每次内容迭代都修改：

- Git 文件名；
- 文档间引用路径；
- Git 历史中的文件重命名；
- 其他工具或脚本中的路径。

---

### 5.2 文档内部元信息与内容版本

需要作为并行专项共同基线的累计文档，应在标题后记录文档元信息。

推荐格式：

```markdown
## 文档元信息

- 文档标识：`tie-build-brainstorm`
- Repository 文件：`tie-build-brainstorm.md`
- 内容版本：`v06`
- 最后更新：`2026-07-12`
- 当前状态：Chat D 候选设计累计文档
```

其中：

- **文档标识**用于跨文件名和跨上传识别同一逻辑文档；
- **Repository 文件**记录仓库中的稳定文件名；
- **内容版本**表示可供一批并行专项共同使用的内容基线；
- **最后更新**帮助人工判断状态是否过期；
- **当前状态**说明文档用途。

内容版本不需要因每次小修改都增加。

建议只在以下情况下增加内容版本：

- 合并完成一批专项成果；
- 文档将成为下一批并行专项的新共同基线；
- 发生会改变专项理解的设计内容修改；
- 需要明确区分两个不能混用的稳定快照。

以下情况通常不必增加内容版本：

- 修正错字；
- 调整排版；
- 不改变含义的措辞优化；
- 更新 Project 自动显示名。

---

### 5.3 ChatGPT Project 显示名与自动编号

文件上传到 ChatGPT Project 后，显示名称可能自动增加编号，例如：

```text
tie-build-brainstorm(6).md
project-workflow-and-handoff(1).md
```

该自动编号：

- 只用于 Project Sources 中的显示与定位；
- 不表示文档内容版本；
- 不需要用户手动改名；
- 可能在删除旧文件并重新上传后继续增加。

因此：

```text
(6) ≠ v06
```

共同基线以：

```text
文档标识 + 文档内部内容版本
```

为准。

任务包和状态快照可以记录当前 Project 显示名，帮助专项会话定位文件；但显示名不是 source of truth。

对于更新频率较低、通常不作为并行内容基线的稳定文档，例如：

```text
Docs/project-workflow-and-handoff.md
```

不要求为了 Project 自动编号而手动维护 `v01`、`v02` 等版本名。

这类文档可以通过以下信息识别：

- 文档标题；
- Repository 路径；
- 文档职责；
- 当前 Project 显示名。

---

### 5.4 专项输出命名

推荐：

```text
<archetype>-chapter-NN-<topic>.md
```

例如：

```text
tie-build-chapter-07-low-hp-sustain.md
tie-build-chapter-08-control.md
```

工程或实现专项可采用对应模块名或批次名。

专项输出在合并前是独立成果，不应冒充共同累计文档。

---

### 5.5 当前 Project 中只保留有效文件

Project Sources 中建议只保留：

- 当前有效累计文档；
- 当前工作所需 canonical 文档；
- 当前有效工作流文档；
- 当前尚未合并的必要专项输出。

旧版本由用户本地或 Git 管理。

如果 Project 中同时存在同一文档标识的多个文件，主协调会话和专项会话必须根据文档内部内容版本选择任务包指定的版本，不能只凭自动编号大小猜测新旧。

# 8. Chat D 工作协议 `[Chat D Only]`

本节只适用于 Chat D 主协调会话和 Chat D 专项会话。

其他 Chat 不应默认套用本节中的章节式讨论流程。

## 8.1 Chat D 的整体职责

Chat D 负责长期玩法头脑风暴。

Chat D 应：

- 打开长期玩法设计空间；
- 探索流派、修正、物品、角色被动、敌人规则和模式规则；
- 分析玩法身份、构筑取舍和系统联动；
- 发现新的设计轴和术语需求；
- 记录复杂度、规则风险、UI 成本和实现风险；
- 为 Chat A 提供可筛选的候选设计材料；
- 保留有长期价值但不适合近期版本的方案。

Chat D 不负责：

- 决定候选是否进入近期实现版本；
- 完成最终数值平衡；
- 擅自修改 locked gameplay rules；
- 编写工程实现规格；
- 拆分 Codex 实现任务；
- 因实现方便而提前删除有长期设计价值的候选。

Chat D 可以指出：

```text
实现成本高
需要新状态系统
需要新 UI
违反当前结算阶段
更适合后续版本
```

但这些判断不等同于最终实现决定。

## 8.2 Chat D 会话类型

### 8.2.1 Chat D 主协调会话

简称：

```text
D主
```

职责：

- 规划当前流派或设计主题；
- 确定章节或专项目录；
- 判断哪些专项适合并行；
- 冻结共同累计文档版本；
- 为每个 D专项生成任务包；
- 维护跨章节、跨流派边界；
- 接收 D专项完成报告；
- 统一检查重复、冲突和遗漏；
- 决定是否需要返工或补充专项；
- 规划下一批专项；
- 在当前聊天失效后，通过本文档与状态快照恢复协调工作。

D主不是正式玩法决策者。Chat D 产出的候选仍需 Chat A 评估。

### 8.2.2 Chat D 专项会话

简称：

```text
D专项
```

职责：

- 只负责指定章节或设计专题；
- 按 Chat D 内部阶段流程与用户多轮讨论；
- 继承任务包中指定的共同基线和边界；
- 不修改其他章节；
- 不重新规划整个流派；
- 不将候选自动提升为正式规则；
- 完成后生成独立 Markdown 文件；
- 返回结构化完成报告。

## 8.3 D主与D专项的协作流程

### 阶段 1：D主规划专项批次

D主应确定：

- 当前流派；
- 当前累计文档；
- 已完成章节；
- 待讨论章节；
- 哪些章节可并行；
- 哪些章节有强依赖，不应并行；
- 各专项之间的主要边界。

### 阶段 2：冻结共同基线

D主明确本批次共同内容基线。

例如：

```text
文档标识：
tie-build-brainstorm

Repository 文件：
tie-build-brainstorm.md

内容版本：
v06

当前 Project 显示名：
tie-build-brainstorm(6).md
```

D主应验证累计文档内部元信息与以上记录一致。

该批次完成前：

- 不改变基线设计内容；
- 不切换到其他内容版本；
- 不把 Project 自动编号当作版本号；
- 不因重新上传产生的新显示名而静默替换基线。
### 阶段 3：D主生成专项任务包

每个 D专项拥有独立任务包。

任务包使用第 8.4 节模板。

### 阶段 4：用户创建 D专项

用户在同一 Project 中新建独立聊天，并粘贴任务包。

D专项读取指定文档后开始工作。

### 阶段 5：D专项独立推进

D专项自行完成：

```text
设计方向分析
→ 用户反馈
→ 方向修订
→ 候选修正
→ 用户反馈
→ 最终章节文件
```

不需要每一步回到 D主汇报。

### 阶段 6：D专项返回完成报告

D专项完成后，使用第 8.5 节模板。

用户将报告或核心结论带回 D主。

### 阶段 7：D主进行跨专项审查

D主检查：

- 重复候选；
- 术语不一致；
- 规则冲突；
- 章节归属冲突；
- 新设计轴；
- 通用规范是否需要更新；
- 是否需要让某个 D专项补充内容。

## 8.6 D专项内部阶段流程

### 阶段 1：设计方向分析

D专项先讨论：

- 本章核心体验；
- 与其他章节的边界；
- 可展开的主要方向；
- 关键规则问题；
- 主要风险；
- 推荐优先级。

此阶段：

- 不直接生成大量完整候选；
- 不生成最终章节文件；
- 不进行最终数值平衡。

### 阶段 2：用户反馈与方向修订

用户通常会逐项评价：

- 哪个方向有趣；
- 哪个风险可以接受；
- 哪个理解需要修正；
- 哪个方向保留、删除或合并；
- 哪条规则应明确锁定。

D专项应：

1. 按用户编号逐项回应；
2. 明确接受、部分接受或不同意的原因；
3. 说明对设计结构造成的改变；
4. 更新方向定义；
5. 总结修订后的方向集合；
6. 不急于生成最终章节。

### 阶段 3：具体候选修正

方向确认后，D专项生成具体候选。

候选通常包括：

- 效果概述；
- 流派分类；
- 触发条件；
- 作用对象；
- 来源与类别；
- 作用时机；
- 持续时间；
- 关键规则；
- 设计意图；
- 联动；
- 风险；
- 相近方案。

候选数值只是讨论基线，不代表正式平衡结果。

### 阶段 4：候选审阅与修订

用户可以：

- 修改机制；
- 合并相似候选；
- 删除候选；
- 增加边界；
- 把部分问题留给 Chat A。

D专项应更新候选结构。

不能因为数值尚未最终平衡，就拒绝继续生成章节。

### 阶段 5：生成替换章节

只有用户明确要求：

```text
生成新的章节
生成文档
整理成章节
请继续生成章节
```

时，才生成最终章节文件。

最终输出应：

- 只包含当前章节；
- 从当前一级标题开始；
- 在下一个一级标题之前结束；
- 不重新生成累计文档；
- 保留已确认结论；
- 将平衡问题明确留给 Chat A；
- 使用独立 Markdown 文件；
- 给出准确替换范围。

## 8.7 “请继续”的解释规则

“请继续”表示进入当前已约定流程的下一个阶段，而不是跳过全部中间步骤。

典型解释：

```text
方向讨论完成后“请继续”
→ 进入具体候选修正阶段。

候选完成并已审阅后“请继续”
→ 如果此前已明确下一步是生成章节，则生成章节。

章节完成后“请继续”
→ 进入下一章方向分析，或返回 D主规划下一专项。
```

如果前一步没有明确下一步，D专项应按既有流程选择最自然阶段。

不得自动：

- 生成整份累计文档；
- 进入工程实现；
- 把候选视为正式规则；
- 跳过用户审阅。

## 8.8 用户反馈处理方式

用户经常使用编号逐项反馈。

D专项应：

- 按编号逐项回应；
- 避免把多个意见合成笼统总结；
- 不只回答“同意”；
- 说明反馈如何改变方向、规则或候选；
- 在回应末尾给出修订后的结构；
- 将纯平衡问题记录后留给 Chat A；
- 对机制变化同步更新后续候选和最终章节。

原则：

> 用户对玩法目标和体验的明确判断，优先于 Chat D 先前提出的候选结构。

## 8.9 玩法结构与平衡问题的分界

Chat D 应重点讨论：

- 是否产生真实决策；
- 是否与其他流派区分；
- 是否形成构筑身份；
- 是否利用 BR3 三轮、三槽位和移位结构；
- 是否违反轮内结算阶段；
- 是否存在触发歧义；
- 是否适合物品、角色被动或其他载体；
- 数值曲线的结构是否支持目标体验。

通常留给 Chat A：

- 最终伤害数值；
- 最终治疗数值；
- 稀有度；
- 出现概率；
- 同名叠加；
- 最终候选池规模；
- 进入哪个版本。

但以下仍属于 Chat D：

```text
第一次不奖励，第二次开始递增
前期低收益，后期高边际收益
需要每场一次上限才能保持玩法身份
```

因为这些是机制结构，而不只是数值微调。

## 8.10 跨章节重叠与归属

每个 D专项开始时，应先判断：

- 哪些内容已经在此前章节讨论；
- 本专项的新触发条件或构筑身份是什么；
- 哪些候选应引用其他章节；
- 某个设计的主要归属在哪里。

处理方式：

1. 指定主要归属章节；
2. 次要章节只简短引用；
3. 明确两个判定条件的区别；
4. 如果无法判断，交回 D主。

示例：

```text
平局 + 吸血牌身份
→ 特质章节。

平局 + 玩家低血状态
→ 低血 / 续航章节。

平局 + 被压制的敌方牌
→ 压制 / 控制章节。
```

## 8.11 新设计轴与术语

如果讨论中发现一个概念可复用于多个流派，应提出更新：

- glossary.md；
- global-modifier-brainstorming-guidelines.md；
- 或相应 canonical 文档。

不应只把通用概念埋在某个章节中。

D专项应在完成报告中记录：

```text
新发现的通用设计轴或术语
```

由 D主决定是否创建单独更新任务。

## 8.12 输出详细度

Chat D 输出应保留足够设计推理，供 Chat A 后续筛选。

方向和候选通常应说明：

- 核心体验；
- 设计价值；
- 主要风险；
- 规则边界；
- 与其他方向的区别；
- 构筑冲突与联动；
- 当前优先级。

应避免：

- 重复整套基础规则；
- 每个候选重复所有通用设计轴；
- 把最终章节写成聊天记录；
- 为了显得完整而生成大量同质候选。

## 8.13 文档生成与合并规则

Chat D 专属规则：

- 累计文档由用户本地维护；
- D专项不直接覆盖累计文档；
- 每次只生成单章或单专项文件；
- 文件名包含流派、章节编号和主题；
- 输出后明确替换起点和下一个一级标题；
- 并行专项完成前不修改共同基线；
- 用户集中合并；
- 合并后上传新版本；
- D主确认新版本后开始下一批。

## 8.14 开放问题与 Chat A handoff

每章结束时，应区分：

### 已确认设计结构

例如：

- 某方向正式保留；
- 某定义采用严格多数；
- 顺序型事件按槽位 1 → 2 → 3 读取。

### 尚未平衡问题

例如：

- 最终伤害数值；
- 稀有度；
- 同名叠加；
- 触发上限。

### 高风险规则问题

例如：

- 改变基础平局定义；
- 产生牌源伤害循环；
- 引入新护盾系统；
- 引入跨战斗状态。

最终章节应明确记录后两类问题，并交给 Chat A。

## 12. 文档维护规则 `[Shared]`

### 12.1 只保存稳定流程和当前快照

本文档应保存：

- 已稳定的协作方式；
- handoff 模板；
- 恢复流程；
- 当前简短状态。

不应保存：

- 每次聊天的完整过程；
- 大量历史任务日志；
- 已经过期的每日状态；
- 可从 canonical 文档直接获取的规则全文。

### 12.2 谁负责更新哪一部分

```text
[Shared]
由相关主协调会话共同维护。

[Chat A Only]
由 Chat A 补充和维护。

[Chat B Only]
由 Chat B 补充和维护。

[Chat C Only]
由 Chat C 补充和维护。

[Chat D Only]
由 Chat D 补充和维护。
```

其他角色不应静默改写某个角色的专属工作协议。

### 12.3 更新时机

建议在以下时机更新：

- 工作方式发生明显变化；
- 新的主协调 / 专项结构被采用；
- 新的 handoff 模板被确认；
- 某个 Chat 角色总结出稳定沟通规则；
- 当前主会话需要恢复；
- 一个流派或实现阶段完成；
- 当前状态快照明显过期。

### 12.4 文档目标

当任意主会话失效时，用户应能依靠：

```text
Docs/project-workflow-and-handoff.md
+ 当前 source-of-truth 文档
+ 当前累计文档
+ 当前状态快照
```

恢复新的主协调会话，并继续工作，而不需要重新摸索协作方式。
