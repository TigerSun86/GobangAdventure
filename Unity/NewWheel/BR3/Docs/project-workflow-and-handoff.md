# BR3 Project Workflow and Handoff

## 1. 文档目的与适用范围 `[Shared]`

本文档用于记录 BR3 项目中，用户与多个 ChatGPT 会话之间已经形成的协作方式、职责边界、交接规则、恢复流程和文件管理规范。

本文档的目标不是定义游戏规则本身，而是确保：

- 不同会话知道自己负责什么；
- 用户与各会话之间的工作方式能够复用；
- 一个会话失效后，可以依靠文档恢复其职责与当前工作状态；
- 多个会话可以围绕独立工作范围并行推进；
- 设计、工程和实现之间的交接保持清晰；
- 聊天记录不是唯一的工作流知识来源。

本文档不是：

- canonical gameplay 规则文档；
- canonical engineering 设计文档；
- 实现规格；
- 临时聊天记录的完整归档；
- Project Instructions 的替代品。

如果本文件引用了玩法、工程或实现规则，应优先引用对应 source-of-truth 文档，而不是在本文件中复制完整规则。

## 2. 术语与命名 `[Shared]`

### 2.1 Chat 角色

BR3 项目当前使用以下会话角色：

- **Chat A**：Gameplay Design
- **Chat B**：Engineering Design
- **Chat C**：Implementation Coordination
- **Chat D**：Gameplay Brainstorming

各角色的详细职责由各自专属工作协议定义。

### 2.2 主协调会话

**主协调会话** 指某个 Chat 角色中负责总体规划、任务拆分、交接、状态维护和结果复查的主要会话。

正式称呼示例：

```text
Chat D 主协调会话
Chat B 主协调会话
```

常用简称：

```text
D 主会话
B 主会话
```

极简简称：

```text
D主
B主
```

### 2.3 专项会话

**专项会话** 指负责一个边界明确、可以独立推进的工作范围的会话。

专项范围可以是：

- 一个玩法章节；
- 一个流派主题；
- 一个平衡专题；
- 一个程序模块；
- 一份工程文档；
- 一个实现批次；
- 一组 Codex 任务；
- 一个问题调查；
- 其他适合独立处理的工作单元。

正式称呼示例：

```text
Chat D 专项会话
Chat B 专项会话
```

常用简称：

```text
D 专项会话
B 专项会话
```

极简简称：

```text
D专项
B专项
```

具体实例可以称为：

```text
第 7 章专项
奖励模块专项
Batch 3 专项
```

### 2.4 不使用“子 Chat”或“子 agent”

专项会话是用户手动创建的独立聊天，不是主协调会话自动创建、调度和回收的技术意义上的 subagent。

因此，不使用以下称呼：

```text
Chat D 子
D子
子 agent
subagent
```

以避免与真正的 agent delegation 混淆。

## 3. Source of Truth 与冲突处理 `[Shared]`

### 3.1 基本原则

聊天记录可以提供上下文，但不是稳定的 source of truth。

稳定知识应优先写入：

- canonical gameplay 文档；
- canonical engineering 文档；
- ADR / GDR；
- 全局规范文档；
- 当前流派累计文档；
- 本工作流、handoff 与跨 Chat 设计方法文档；
- 明确的当前状态快照。

### 3.2 Gameplay 层级

Gameplay 内容建议按以下优先级理解：

```text
locked / canonical gameplay documents
> Chat A 已接受并写入正式文档的设计
> Chat D 候选设计文档
> 临时 handoff
> 单次聊天中的未落盘讨论
```

Chat D 中出现的候选不自动进入正式实现范围。

### 3.3 Engineering 层级

Engineering 内容建议按以下优先级理解：

```text
canonical engineering documents / ADR
> Chat B 已确认的工程 handoff
> Chat C implementation plan
> Codex 临时解释或建议
```

Chat C 和 Codex 不应静默修改已接受的玩法或工程设计。

### 3.4 冲突处理

发现冲突时：

1. 不静默选择自己偏好的版本；
2. 确认冲突涉及 gameplay、engineering 还是 implementation；
3. 回到对应负责角色处理；
4. 更新 source-of-truth 文档；
5. 再继续后续工作。

典型归属：

```text
玩法取舍或规则冲突 → Chat A
长期候选空间与流派讨论 → Chat D
架构、领域模型、流程、测试设计 → Chat B
实现批次、Codex 提示词、落地顺序 → Chat C
```

## 4. 通用多会话协作原则 `[Shared]`

### 4.1 主协调会话与专项会话分工

主协调会话负责：

- 规划工作范围；
- 判断哪些任务适合拆分；
- 冻结本批次共同输入；
- 生成专项任务包；
- 维护跨专项边界；
- 接收完成报告；
- 统一检查重复、冲突和遗漏；
- 规划下一批工作。

专项会话负责：

- 只处理被分配的范围；
- 按对应角色的工作协议推进；
- 不自行扩大职责；
- 不修改其他专项内容；
- 完成后返回结构化报告。

### 4.2 固定协议与动态任务包分离

固定工作方式保存在本文档。

每次创建专项会话时，主协调会话只需要生成一份动态任务包，填写当前任务的变量。

动态任务包至少应包含：

- 角色；
- 负责范围；
- 当前阶段；
- 共同基线；
- 必读文档；
- 已确认边界；
- 相关前置工作；
- 输出范围；
- 目标文件名；
- 完成后回传要求。

不建议只写：

```text
参见 handoff 文档第 X 节，负责某任务。
```

因为这不足以说明当前基线、任务阶段和专项边界。

### 4.3 冻结共同基线

同一批并行专项开始前，主协调会话应明确一个共同内容基线。

共同基线应主要由以下信息确定：

```text
文档标识
+ 文档内部内容版本
```

Repository 文件名用于说明文档在仓库中的稳定位置；ChatGPT Project 显示名只用于在 Project Sources 中定位当前上传文件。

例如：

```text
共同基线：

文档标识：
tie-build-brainstorm

Repository 文件：
tie-build-brainstorm.md

内容版本：
v06

当前 Project 显示名：
tie-build-brainstorm(6).md
```

开始专项前，主协调会话应确认：

- Project 中读取到的文档标识正确；
- 文档内部内容版本与任务包一致；
- 当前 Project 显示名指向该内容版本；
- ChatGPT 自动添加的 `(N)` 后缀没有被误认为内容版本。

在该批专项完成前：

- 不改变共同基线的设计内容；
- 不让不同专项分别使用不同内容版本；
- 不在中途把某个专项结果直接合并后重新上传给其他专项；
- 不因 Project 显示名变化而静默切换基线；
- 等整批完成后再集中合并，并在需要时更新内部内容版本。

### 4.4 专项不直接修改共同累计文档

专项会话应只输出自己的独立工作成果。

例如：

```text
tie-build-chapter-07-low-hp-sustain.md
```

而不是重新生成：

```text
tie-build-brainstorm.md
```

共同累计文档由用户在本地集中维护和合并。

### 4.5 聊天记录不是唯一依赖

专项会话应主要依赖：

- source-of-truth 文档；
- 当前共同基线；
- 本工作流文档；
- 主协调会话生成的任务包。

不应依赖：

```text
请自己在 Project 中找出主会话之前说过什么。
```

项目内聊天召回可以作为辅助，但不能代替明确的文档和 handoff。

## 5. 文件与版本管理 `[Shared]`

### 5.1 Repository 中使用稳定文件名

Repository 中的累计设计文档应优先使用稳定文件名：

```text
<archetype>-brainstorm.md
```

例如：

```text
tie-build-brainstorm.md
movement-build-brainstorm.md
```

这样可以避免每次内容迭代都修改：

- Git 文件名；
- 文档间引用路径；
- Git 历史中的文件重命名；
- 其他工具或脚本中的路径。

---

### 5.2 文档内部元信息与内容版本

需要作为并行专项共同基线的累计文档，应在标题后记录文档元信息。

推荐格式：

```markdown
## 文档元信息

- 文档标识：`tie-build-brainstorm`
- Repository 文件：`tie-build-brainstorm.md`
- 内容版本：`v06`
- 最后更新：`2026-07-12`
- 当前状态：Chat D 候选设计累计文档
```

其中：

- **文档标识**用于跨文件名和跨上传识别同一逻辑文档；
- **Repository 文件**记录仓库中的稳定文件名；
- **内容版本**表示可供一批并行专项共同使用的内容基线；
- **最后更新**帮助人工判断状态是否过期；
- **当前状态**说明文档用途。

内容版本不需要因每次小修改都增加。

建议只在以下情况下增加内容版本：

- 合并完成一批专项成果；
- 文档将成为下一批并行专项的新共同基线；
- 发生会改变专项理解的设计内容修改；
- 需要明确区分两个不能混用的稳定快照。

以下情况通常不必增加内容版本：

- 修正错字；
- 调整排版；
- 不改变含义的措辞优化；
- 更新 Project 自动显示名。

---

### 5.3 ChatGPT Project 显示名与自动编号

文件上传到 ChatGPT Project 后，显示名称可能自动增加编号，例如：

```text
tie-build-brainstorm(6).md
project-workflow-and-handoff(1).md
```

该自动编号：

- 只用于 Project Sources 中的显示与定位；
- 不表示文档内容版本；
- 不需要用户手动改名；
- 可能在删除旧文件并重新上传后继续增加。

因此：

```text
(6) ≠ v06
```

共同基线以：

```text
文档标识 + 文档内部内容版本
```

为准。

任务包和状态快照可以记录当前 Project 显示名，帮助专项会话定位文件；但显示名不是 source of truth。

对于更新频率较低、通常不作为并行内容基线的稳定文档，例如：

```text
Docs/project-workflow-and-handoff.md
```

不要求为了 Project 自动编号而手动维护 `v01`、`v02` 等版本名。

这类文档可以通过以下信息识别：

- 文档标题；
- Repository 路径；
- 文档职责；
- 当前 Project 显示名。

---

### 5.4 专项输出命名

推荐：

```text
<archetype>-chapter-NN-<topic>.md
```

例如：

```text
tie-build-chapter-07-low-hp-sustain.md
tie-build-chapter-08-control.md
```

工程或实现专项可采用对应模块名或批次名。

专项输出在合并前是独立成果，不应冒充共同累计文档。

---

### 5.5 当前 Project 中只保留有效文件

Project Sources 中建议只保留：

- 当前有效累计文档；
- 当前工作所需 canonical 文档；
- 当前有效工作流文档；
- 当前尚未合并的必要专项输出。

旧版本由用户本地或 Git 管理。

如果 Project 中同时存在同一文档标识的多个文件，主协调会话和专项会话必须根据文档内部内容版本选择任务包指定的版本，不能只凭自动编号大小猜测新旧。

# 8. Chat D 工作协议 `[Chat D Only]`

本节只适用于 Chat D 主协调会话和 Chat D 专项会话。

其他 Chat 不应默认套用本节中的章节式讨论流程。

## 8.1 Chat D 的整体职责

Chat D 负责长期玩法头脑风暴。

Chat D 应：

- 打开长期玩法设计空间；
- 探索流派、修正、物品、角色被动、敌人规则和模式规则；
- 分析玩法身份、构筑取舍和系统联动；
- 发现新的设计轴和术语需求；
- 记录复杂度、规则风险、UI 成本和实现风险；
- 为 Chat A 提供可筛选的候选设计材料；
- 保留有长期价值但不适合近期版本的方案。

Chat D 不负责：

- 决定候选是否进入近期实现版本；
- 完成最终数值平衡；
- 擅自修改 locked gameplay rules；
- 编写工程实现规格；
- 拆分 Codex 实现任务；
- 因实现方便而提前删除有长期设计价值的候选。

Chat D 可以指出：

```text
实现成本高
需要新状态系统
需要新 UI
违反当前结算阶段
更适合后续版本
```

但这些判断不等同于最终实现决定。

## 8.2 Chat D 会话类型

### 8.2.1 Chat D 主协调会话

简称：

```text
D主
```

职责：

- 规划当前流派或设计主题；
- 确定章节或专项目录；
- 判断哪些专项适合并行；
- 冻结共同累计文档版本；
- 为每个 D专项生成任务包；
- 维护跨章节、跨流派边界；
- 接收 D专项完成报告；
- 统一检查重复、冲突和遗漏；
- 决定是否需要返工或补充专项；
- 规划下一批专项；
- 在当前聊天失效后，通过本文档与状态快照恢复协调工作。

D主不是正式玩法决策者。Chat D 产出的候选仍需 Chat A 评估。

### 8.2.2 Chat D 专项会话

简称：

```text
D专项
```

职责：

- 只负责指定章节或设计专题；
- 按 Chat D 内部阶段流程与用户多轮讨论；
- 继承任务包中指定的共同基线和边界；
- 不修改其他章节；
- 不重新规划整个流派；
- 不将候选自动提升为正式规则；
- 完成后生成独立 Markdown 文件；
- 返回结构化完成报告。

## 8.3 D主与D专项的协作流程

### 阶段 1：D主规划专项批次

D主应确定：

- 当前流派；
- 当前累计文档；
- 已完成章节；
- 待讨论章节；
- 哪些章节可并行；
- 哪些章节有强依赖，不应并行；
- 各专项之间的主要边界。

### 阶段 2：冻结共同基线

D主明确本批次共同内容基线。

例如：

```text
文档标识：
tie-build-brainstorm

Repository 文件：
tie-build-brainstorm.md

内容版本：
v06

当前 Project 显示名：
tie-build-brainstorm(6).md
```

D主应验证累计文档内部元信息与以上记录一致。

该批次完成前：

- 不改变基线设计内容；
- 不切换到其他内容版本；
- 不把 Project 自动编号当作版本号；
- 不因重新上传产生的新显示名而静默替换基线。

### 阶段 3：D主生成专项任务包

每个 D专项拥有独立任务包。

任务包使用第 8.4 节模板。

### 阶段 4：用户创建 D专项

用户在同一 Project 中新建独立聊天，并粘贴任务包。

D专项读取指定文档后开始工作。

### 阶段 5：D专项独立推进

D专项自行完成：

```text
设计方向分析
→ 用户反馈
→ 方向修订
→ 候选修正
→ 用户反馈
→ 最终章节文件
```

不需要每一步回到 D主汇报。

### 阶段 6：D专项返回完成报告

D专项完成后，使用第 8.5 节模板。

用户将报告或核心结论带回 D主。

### 阶段 7：D主进行跨专项审查

D主检查：

- 重复候选；
- 术语不一致；
- 规则冲突；
- 章节归属冲突；
- 新设计轴；
- 通用规范是否需要更新；

## 8.6 D专项内部阶段流程

### 阶段 1：设计方向分析

D专项先讨论：

- 本章核心体验；
- 与其他章节的边界；
- 可展开的主要方向；
- 关键规则问题；
- 主要风险；
- 推荐优先级。

此阶段：

- 不直接生成大量完整候选；
- 不生成最终章节文件；
- 不进行最终数值平衡。

### 阶段 2：用户反馈与方向修订

用户通常会逐项评价：

- 哪个方向有趣；
- 哪个风险可以接受；
- 哪个理解需要修正；
- 哪个方向保留、删除或合并；
- 哪条规则应明确锁定。

D专项应：

1. 按用户编号逐项回应；
2. 明确接受、部分接受或不同意的原因；
3. 说明对设计结构造成的改变；
4. 更新方向定义；
5. 总结修订后的方向集合；
6. 不急于生成最终章节。

### 阶段 3：具体候选修正

方向确认后，D专项生成具体候选。

候选通常包括：

- 效果概述；
- 流派分类；
- 触发条件；
- 作用对象；
- 来源与类别；
- 作用时机；
- 持续时间；
- 关键规则；
- 设计意图；
- 联动；
- 风险；
- 相近方案。

候选数值只是讨论基线，不代表正式平衡结果。

### 阶段 4：候选审阅与修订

用户可以：

- 修改机制；
- 合并相似候选；
- 删除候选；
- 增加边界；
- 把部分问题留给 Chat A。

D专项应更新候选结构。

不能因为数值尚未最终平衡，就拒绝继续生成章节。

### 阶段 5：生成替换章节

只有用户明确要求：

```text
生成新的章节
生成文档
整理成章节
请继续生成章节
```

时，才生成最终章节文件。

最终输出应：

- 只包含当前章节；
- 从当前一级标题开始；
- 在下一个一级标题之前结束；
- 不重新生成累计文档；
- 保留已确认结论；
- 将平衡问题明确留给 Chat A；
- 使用独立 Markdown 文件；
- 给出准确替换范围。

## 8.7 “请继续”的解释规则

“请继续”表示进入当前已约定流程的下一个阶段，而不是跳过全部中间步骤。

典型解释：

```text
方向讨论完成后“请继续”
→ 进入具体候选修正阶段。

候选完成并已审阅后“请继续”
→ 如果此前已明确下一步是生成章节，则生成章节。

章节完成后“请继续”
→ 进入下一章方向分析，或返回 D主规划下一专项。
```

如果前一步没有明确下一步，D专项应按既有流程选择最自然阶段。

不得自动：

- 生成整份累计文档；
- 进入工程实现；
- 把候选视为正式规则；
- 跳过用户审阅。

## 8.8 用户反馈处理方式

用户经常使用编号逐项反馈。

D专项应：

- 按编号逐项回应；
- 避免把多个意见合成笼统总结；
- 不只回答“同意”；
- 说明反馈如何改变方向、规则或候选；
- 在回应末尾给出修订后的结构；
- 将纯平衡问题记录后留给 Chat A；
- 对机制变化同步更新后续候选和最终章节。

原则：

> 用户对玩法目标和体验的明确判断，优先于 Chat D 先前提出的候选结构。

## 8.9 玩法结构与平衡问题的分界

Chat D 应重点讨论：

- 是否产生真实决策；
- 是否与其他流派区分；
- 是否形成构筑身份；
- 是否利用 BR3 三轮、三槽位和移位结构；
- 是否违反轮内结算阶段；
- 是否存在触发歧义；
- 是否适合物品、角色被动或其他载体；
- 数值曲线的结构是否支持目标体验。

通常留给 Chat A：

- 最终伤害数值；
- 最终治疗数值；
- 稀有度；
- 出现概率；
- 同名叠加；
- 最终候选池规模；
- 进入哪个版本。

但以下仍属于 Chat D：

```text
第一次不奖励，第二次开始递增
前期低收益，后期高边际收益
需要每场一次上限才能保持玩法身份
```

因为这些是机制结构，而不只是数值微调。

## 8.10 跨章节重叠与归属

每个 D专项开始时，应先判断：

- 哪些内容已经在此前章节讨论；
- 本专项的新触发条件或构筑身份是什么；
- 哪些候选应引用其他章节；
- 某个设计的主要归属在哪里。

处理方式：

1. 指定主要归属章节；
2. 次要章节只简短引用；
3. 明确两个判定条件的区别；
4. 如果无法判断，交回 D主。

示例：

```text
平局 + 吸血牌身份
→ 特质章节。

平局 + 玩家低血状态
→ 低血 / 续航章节。

平局 + 被压制的敌方牌
→ 压制 / 控制章节。
```

## 8.11 新设计轴与术语

如果讨论中发现一个概念可复用于多个流派，应提出更新：

- glossary.md；
- 本文档第 12.1 节“Chat D 与 Chat A：修正头脑风暴与评估”；
- 或相应 canonical 文档。

不应只把通用概念埋在某个章节中。

D专项应在完成报告中记录：

```text
新发现的通用设计轴或术语
```

由 D主决定是否创建单独更新任务。

## 8.12 输出详细度

Chat D 输出应保留足够设计推理，供 Chat A 后续筛选。

方向和候选通常应说明：

- 核心体验；
- 设计价值；
- 主要风险；
- 规则边界；
- 与其他方向的区别；
- 构筑冲突与联动；
- 当前优先级。

应避免：

- 重复整套基础规则；
- 每个候选重复所有通用设计轴；
- 把最终章节写成聊天记录；
- 为了显得完整而生成大量同质候选。

## 8.13 文档生成与合并规则

Chat D 专属规则：

- 累计文档由用户本地维护；
- D专项不直接覆盖累计文档；
- 每次只生成单章或单专项文件；
- 文件名包含流派、章节编号和主题；
- 输出后明确替换起点和下一个一级标题；
- 并行专项完成前不修改共同基线；
- 用户集中合并；
- 合并后上传新版本；
- D主确认新版本后开始下一批。

## 8.14 开放问题与 Chat A handoff

每章结束时，应区分：

### 已确认设计结构

例如：

- 某方向正式保留；
- 某定义采用严格多数；
- 顺序型事件按槽位 1 → 2 → 3 读取。

### 尚未平衡问题

例如：

- 最终伤害数值；
- 稀有度；
- 同名叠加；
- 触发上限。

### 高风险规则问题

例如：

- 改变基础平局定义；
- 产生牌源伤害循环；
- 引入新护盾系统；
- 引入跨战斗状态。

最终章节应明确记录后两类问题，并交给 Chat A。

## 12.1 Chat D 与 Chat A：修正头脑风暴与评估 `[Chat D + Chat A Only]`

### 12.1.1 目的与适用范围

本节用于指导 BR3 关于以下内容的头脑风暴、整理和评估：

- 修正 / Modifier；

- 模式规则；

### 12.1.3 修正设计轴

本节记录设计修正时可反复使用的通用分析维度。

这些设计轴不是某一个流派专属的方法。它们可以用于平局流、移位流、低血流、反伤流、毒伤流、奖励经济流等后续所有流派讨论。

D专项在头脑风暴时，可以用这些设计轴打开当前专项范围内的设计空间。D主可用这些设计轴检查跨专项覆盖是否完整；A主在评估时，可以用这些设计轴检查候选修正是否存在重复、歧义、时序风险、UI 风险或平衡风险。

---

#### 12.1.3.1 玩家体验轴

玩家体验轴用于回答：

> 这个修正主要让玩家体验到什么？

常见方向包括：

- 输出：把某个行为转化为伤害或击败效率；
- 续航：提供治疗、减伤、护盾、护甲、承伤缓冲；
- 蓄力：把当前行为积累为之后释放的收益；
- 成长：让牌、角色或构筑在更长跨度上变强；
- 控制：削弱敌人、限制敌人、改变敌人后续威胁；
- 阵型：强调槽位、相邻、对位、多槽结构；
- 奖励经济：影响奖励生成、奖励质量、奖励选择或构筑收束；
- 规则改写：改变基础规则本身的含义。

规则改写型通常风险较高，应特别谨慎。它适合少量核心被动、高稀有修正、敌人规则或模式规则，不应轻易作为普通物品词条大量出现。

---

#### 12.1.3.2 触发组合轴

触发组合轴用于回答：

> 这个修正是只属于一个流派，还是同时要求多个流派条件？

常见类型包括：

- 纯流派修正：只要求一个核心条件；
- 混合触发修正：触发条件同时包含两个或更多流派概念；
- 涌现联动：多个独立修正自然组合后产生额外效果。

示例：

```text
纯流派修正：
发生槽位平局时，敌人受到 1 点保底伤害。

混合触发修正：
槽位 1 发生槽位平局时，敌人受到 2 点保底伤害。

涌现联动：
一个修正让槽位平局造成保底伤害，另一个修正让保底伤害 +1。
```

设计讨论时，应系统展开混合触发修正，但不需要穷尽所有涌现联动。涌现联动只记录特别有趣或特别危险的例子。

---

#### 12.1.3.3 来源轴

来源轴用于回答：

> 这个效果归属于谁？

当前最重要的是：

- 伤害源；
- 治疗源。

候选伤害源包括：

- 牌；
- 槽位；
- 物品；
- 角色；
- 敌人；
- 系统。

候选治疗源包括：

- 牌；
- 物品；
- 角色；
- 敌人；
- 系统。

默认原则：

- 牌源伤害可以触发该牌自身的伤害相关特质或效果；
- 非牌源伤害默认不触发牌特质，除非修正特别说明；
- 牌源治疗可以触发该牌自身的治疗相关效果；
- 非牌源治疗默认不触发牌特质，除非修正特别说明。

未来如果需要更细的统计或联动，可以继续讨论强化源、削弱源、控制源等概念。但在当前阶段，伤害源和治疗源优先级最高。

---

#### 12.1.3.4 伤害类别轴

伤害类别轴用于回答：

> 这段伤害属于哪一类规则？

当前建议伤害类别包括：

- 槽位战斗伤害；
- 保底伤害；
- 反伤；
- 毒伤；
- 治疗转伤；
- 固定伤害；
- 系统伤害。

这些类别目前是设计讨论中的建议列表，不是已锁定的实现枚举。

设计原则：

> 不要为每个触发条件创建新的伤害类别。

例如，通常不应创建：

- 平局固定伤害；
- 移位固定伤害；
- 治疗固定伤害；
- 槽位 1 固定伤害。

更推荐将“为什么触发”和“伤害类别”分开：

```text
触发条件：发生槽位平局
伤害类别：保底伤害
```

```text
触发条件：发生移位
伤害类别：固定伤害
```

触发条件说明为什么发生，伤害类别说明这段伤害在规则系统中属于哪类。

---

#### 12.1.3.5 作用时机轴

作用时机轴用于回答：

> 这个修正在什么时候结算？

常见作用时机包括：

- 指定轮内结算阶段开始时；
- 指定轮内结算阶段结束时；
- 槽位结果产生后；
- 合并伤害阶段后；
- 结算后阶段；
- 本轮结束时；
- 下一轮开始时；
- 战斗结束时；
- 击败敌人时；
- 生成奖励时；
- 选择奖励时。

作用时机必须尊重当前轮内结算阶段顺序。

尤其要避免：

> 在槽位结果已经产生后，回头修改本轮已经参与槽位结算的强度、胜负或槽位结果。

如果某个修正在槽位结果产生后触发，通常应改为：

- 影响下一轮；
- 造成额外伤害；
- 产生治疗；
- 记录状态；
- 影响后续奖励；
- 影响后续战斗。

---

#### 12.1.3.6 持续时间与释放跨度轴

持续时间与释放跨度轴用于回答：

> 这个修正的效果持续到什么时候？它在多长的时间跨度后释放收益？

常见持续时间包括：

| 持续时间 | 说明 | 设计倾向 |
|---|---|---|
| 本次触发 | 只在当前触发事件中生效 | 最简单，风险低 |
| 本轮 | 当前轮内生效 | 需要注意轮内阶段顺序 |
| 下一轮 | 下一轮生效或释放 | 很适合短期蓄力 |
| 固定轮次 | 例如第 3 轮释放 | 适合利用 BR3 的三轮结构 |
| 本场战斗 | 本场战斗内保留或累计 | 构筑感较强，需要展示 |
| 当前敌人阶段 / 跨战斗 | 同一个敌人的多场战斗之间保留 | 长线经营感强，数值应更高，数量不宜多 |
| 一局游戏 / 跨敌人 | 可以带到后续敌人 | 接近 run 内永久收益，风险较高 |
| 跨局游戏 | 下一局游戏仍然保留 | 属于局外成长，早期版本不考虑 |

一般原则：

- 持续时间越长，通常越需要更高收益来补偿延迟满足；
- 持续时间越长，越需要 UI 或日志清楚展示；
- 持续时间越长，越容易产生拖延战斗、长期数值失控或记忆负担；
- 跨战斗和跨敌人修正应少量使用，并交给 Chat A 重点评估。

---

#### 12.1.3.7 释放条件轴

释放条件轴用于回答：

> 已经积累的收益在什么条件下兑现？

常见释放条件包括：

- 自动释放；
- 下一轮释放；
- 第 N 轮释放；
- 本轮结束释放；
- 战斗结束释放；
- 槽位胜利释放；
- 槽位失败释放；
- 受到伤害释放；
- 造成治疗释放；
- 低血时释放；
- 生成奖励时释放；
- 选择某类奖励时释放。

设计时应特别注意：

1. 若释放条件依赖多个槽位同时结算，例如“下一次槽位胜利”，需要明确槽位顺序或改成无顺序歧义的写法。
2. 若释放条件过于隐蔽，需要额外 UI / 日志支持。
3. 若释放条件过于稳定，可能变成无脑收益。
4. 若释放条件过难，玩家可能感受不到修正价值。

示例：

```text
平局积累，槽位胜利释放：
每发生 1 次槽位平局，获得 1 层均势。
下一次槽位胜利时，消耗均势并造成额外伤害。
```

这种设计可以避免玩家完全放弃槽位胜利，让平局成为准备动作，槽位胜利成为释放动作。

---

#### 12.1.3.8 可见性与 UI 成本轴

可见性与 UI 成本轴用于回答：

> 玩家是否能看懂这个修正已经触发、正在持续、即将释放或已经释放？

常见可见性需求包括：

- 只需要战斗日志；
- 需要状态图标；
- 需要状态图标上的层数数字；
- 需要版面标记；
- 需要槽位标记；
- 需要下一轮预览；
- 需要奖励生成解释；
- 需要战斗结束汇总。

设计原则：

- 即时伤害、即时治疗等效果，通常只需要日志和数值反馈；
- 下一轮生效的状态，至少需要日志，正式 UI 中最好有状态提示；
- 层数型资源需要能显示当前层数；
- 跨战斗或跨敌人状态需要更明确的持久状态展示；
- 奖励经济类修正如果改变奖励生成，应尽量解释“为什么这次奖励变了”。

D专项可以先记录 UI 需求和风险，不需要完成 UI 方案。候选进一步被接受后，再由相应玩法、工程或 UI 会话处理具体方案。

---

#### 12.1.3.9 数值与限制轴

数值与限制轴用于回答：

> 这个修正如何防止过强、过弱或无限滚雪球？

常见限制方式包括：

- 无额外限制；
- 每轮最多 1 次；
- 每场战斗最多 1 次；
- 每张牌每场战斗最多 1 次；
- 同名修正不叠加；
- 层数上限；
- 只在第 2 / 第 3 轮生效；
- 只在敌人未被击败时生效；
- 只在玩家低血时生效；
- 只影响下一次触发；
- 战斗结束清空；
- 当前敌人阶段结束清空。

如果没有额外限制，应写“无”，而不是重复写规则本身已经隐含的限制。

例如某个槽位每轮本来只会产生一次槽位结果，则不必写“每个槽位每轮最多触发 1 次”，除非该限制用于处理同名修正叠加或其他特殊情况。

---

#### 12.1.3.10 设计轴使用建议

D专项头脑风暴时，不需要对每个修正都完整遍历所有设计轴。

建议用法：

1. 先用玩家体验轴确定这个修正想带来的体验；
2. 再用触发组合轴判断它是纯流派修正、混合触发修正，还是涌现联动；
3. 如果涉及伤害或治疗，再检查来源轴与伤害类别轴；
4. 如果效果延后释放，再检查持续时间与释放条件；
5. 如果状态需要记忆或累计，再检查可见性与 UI 成本；
6. 如果可能过强或重复触发，再检查数值与限制轴；
7. 最后检查是否违反轮内结算阶段顺序。

这些设计轴的作用是帮助发现设计空间和风险，不是要求每条修正都写成完整规格。

---


### 12.1.4 修正描述模板

修正可以使用两种模板描述：

- **简写模板**：用于 D专项快速头脑风暴；
- **完整模板**：用于候选修正准备移交 Chat A 或进入较严肃评估时。

#### 12.1.4.1 简写模板

D专项在当前范围内大量发散时，优先使用简写模板。

```markdown
##### 修正名

**效果概述：**
一句话描述这个修正，尽量包含触发条件、作用对象和主要效果。

- 流派：
- 可用载体：
- 关键规则：
- 设计意图：
- 联动备注：
- 风险：
```

##### 适用情况

简写模板适合：

- 初次提出想法；
- 同一流派下快速列举多个方向；
- 尚不确定是否值得深入的修正；
- 不涉及复杂伤害源、治疗源、伤害类别或作用时机的修正。

##### 示例

```markdown
##### 平衡之刃

**效果概述：**
发生槽位平局时，敌人受到 1 点保底伤害。

- 流派：平局流
- 可用载体：物品词条 / 角色被动
- 关键规则：槽位平局触发；伤害类别为保底伤害。
- 设计意图：让平局成为稳定低风险输出来源。
- 联动备注：可与“保底伤害 +1”类修正联动。
- 风险：若保底伤害容易叠高，平局输出可能过强。
```

---

#### 12.1.4.2 完整模板

完整模板用于更严谨地描述候选修正。D专项准备写入最终章节或完成报告、或 D主准备移交 A主评估的重点修正，建议尽量补全完整模板。

```markdown
##### 修正名
```

### 12.1.5 完整模板字段说明

#### 12.1.5.1 效果概述

一句话快速说明修正做什么。

效果概述应尽量包含：

- 触发条件；
- 主要作用对象；
- 主要效果；
- 若涉及伤害，尽量说明伤害类别。

效果概述用于快速查阅，不承担所有规则严谨性。更细节的歧义应在后续字段中消除。

##### 示例

```markdown
**效果概述：**
发生槽位平局时，敌人受到 1 点保底伤害。
```

```markdown
**效果概述：**
拥有吸血的牌发生槽位平局时，该牌对敌人造成 1 点保底伤害。
```

---

#### 12.1.5.2 流派

说明该修正服务于哪些流派。

示例：

- 平局流；
- 移位流；
- 平局流 / 位置流；
- 低血流 / 续航流；
- 奖励经济流。

如果一个修正的触发条件同时包含两个流派概念，应标记为混合触发修正。

---

#### 12.1.5.3 可用载体

说明这个修正可能以什么形式出现。

常见载体包括：

- 物品词条；
- 角色被动；
- 敌人规则；
- 模式规则；
- 未来可能出现的牌修正或槽位修正。

同一个修正原型可以有多个潜在载体。Chat D 中不必过早锁定载体。

---

#### 12.1.5.4 触发条件

说明什么时候检查或触发该修正。

示例：

- 发生槽位平局时；
- 槽位 1 发生槽位平局时；
- 一张牌发生移位后；
- 本轮至少发生 2 次槽位平局时；
- 玩家 HP 低于一半时；
- 战斗结束时；
- 生成下一次奖励时。

触发条件不应混入伤害源、治疗源或作用对象类型。

---

#### 12.1.5.5 作用时机

说明修正在哪个时间点结算。

常见作用时机包括：

- 指定轮内结算阶段开始时；
- 指定轮内结算阶段结束时；
- 槽位结果产生后；
- 合并伤害阶段后；
- 结算后阶段；
- 本轮结束时；
- 战斗结束时；
- 击败敌人时；
- 生成奖励时；
- 选择奖励时。

作用时机必须尊重当前轮内结算阶段顺序。见本文第 5 节。

---

#### 12.1.5.6 作用对象类型

作用对象类型只描述被影响对象的大类，不混入范围信息。

候选对象类型包括：

- 玩家；
- 敌人；
- 牌；
- 槽位；
- 奖励选项；
- 奖励生成规则；
- 物品；
- 角色；
- 系统。

示例：

```markdown
- 作用对象类型：敌人
```

```markdown
- 作用对象类型：牌
```

```markdown
- 作用对象类型：奖励生成规则
```

---

#### 12.1.5.7 作用范围

作用范围说明在哪里或以什么范围选择作用对象。

示例：

- 当前敌人；
- 玩家侧；
- 敌人侧；
- 触发槽位；
- 触发槽位的相邻槽位；
- 对位槽位；
- 所有已开启槽位；
- 下一次奖励报价；
- 当前敌人阶段；
- 整个一局游戏。

作用范围和筛选条件的边界后续可由 Chat A 进一步调整。当前文档先保留两个字段。

---

#### 12.1.5.8 筛选条件

筛选条件用于在作用范围内进一步选择目标。

示例：

- 只影响拥有 `Lifesteal` 的牌；
- 只影响发生槽位平局的槽位；
- 只影响 Upgrade 奖励选项；
- 只影响当前未满特质上限的牌；
- 只影响本轮新打出的牌。

如果没有额外筛选条件，可写“无”。

---

#### 12.1.5.9 效果细则

效果细则用于严谨描述实际效果。

示例：

```markdown
- 效果细则：敌人受到 1 点保底伤害。
```

```markdown
- 效果细则：作用对象下一轮本轮强度 +1。
```

```markdown
- 效果细则：下一次奖励生成时，提高 Upgrade 候选的出现权重。
```

若效果涉及 HP 回复，应说明是否受最大 HP 限制。默认治疗不能超过最大 HP。

---

#### 12.1.5.10 伤害源

当修正造成伤害时，应明确伤害源。

候选伤害源包括：

- 牌；
- 槽位；
- 物品；
- 角色；
- 敌人；
- 系统。

伤害源只描述伤害归属，不描述触发条件。

##### 示例

```markdown
- 伤害源：槽位
```

```markdown
- 伤害源：牌
```

```markdown
- 伤害源：物品
```

默认情况下，只有牌源伤害会触发该牌自身的伤害相关特质或效果，例如吸血。非牌源伤害不触发牌特质，除非修正特别说明。

---

#### 12.1.5.11 伤害类别

当修正造成伤害时，应明确伤害类别。

当前建议伤害类别包括：

- 槽位战斗伤害；
- 保底伤害；
- 反伤；
- 毒伤；
- 治疗转伤；
- 固定伤害；
- 系统伤害。

这些类别目前是设计讨论中的建议列表，不是已锁定的实现枚举。

##### 设计原则

不要为每个触发条件创建新的伤害类别。

例如，通常不应创建：

- 平局固定伤害；
- 移位固定伤害；
- 治疗固定伤害；
- 槽位 1 固定伤害。

更推荐写成：

```markdown
触发条件：发生槽位平局
伤害类别：保底伤害
```

或：

```markdown
触发条件：发生移位
伤害类别：固定伤害
```

触发条件说明为什么发生，伤害类别说明这段伤害属于哪类规则。

---

#### 12.1.5.12 治疗源

当修正产生治疗时，应明确治疗源。

候选治疗源包括：

- 牌；
- 物品；
- 角色；
- 敌人；
- 系统。

默认情况下，只有牌源治疗会触发该牌自身的治疗相关特质或效果。非牌源治疗不触发牌特质，除非修正特别说明。

当前 v1 规则中，所有治疗都不能使 HP 超过最大 HP。

---

#### 12.1.5.13 持续时间

持续时间说明效果持续多久。

示例：

- 本次触发；
- 本轮；
- 下一轮；
- 本场战斗；
- 当前敌人阶段；
- 一局游戏；
- 永久。

不要使用“持续范围”表示 Duration，以免和作用范围混淆。

---

#### 12.1.5.14 限制

限制用于说明额外触发次数、叠加、频率或边界。

示例：

- 无；
- 每轮最多 1 次；
- 每场战斗最多 1 次；
- 每张牌每场战斗最多 1 次；
- 同名修正不叠加；
- 即使拥有多个同名修正，每轮也只触发 1 次。

如果没有额外限制，应写“无”，而不是写规则本身已经隐含的限制。

例如某个槽位每轮本来只会产生一次槽位结果，则不必写“每个槽位每轮最多触发 1 次”，除非该限制是为了处理同名修正叠加或其他特殊情况。

---

#### 12.1.5.15 设计意图

设计意图说明这个修正希望制造什么玩家行为或体验。

示例：

- 让槽位平局成为稳定低风险输出来源；
- 鼓励玩家围绕移位制造后续收益；
- 让低血状态成为可主动利用的风险资源；
- 让奖励选择更容易朝某个流派收束。

设计意图比数值更重要。Chat A 后续可以调整数值，但应尽量保留明确的体验目标。

---

#### 12.1.5.16 联动备注

联动备注用于记录修正可能和其他规则产生的自然联动或设计联动。

示例：

- 与“保底伤害 +1”联动；
- 若伤害源是牌，则可能触发吸血；
- 与溢出治疗转伤联动后可能过强；
- 可与移位流联动，玩家通过移位主动制造槽位平局。

联动备注不需要穷尽所有组合，只记录特别有价值或特别危险的联动。

---

#### 12.1.5.17 相近修正区别

相近修正区别用于说明这个修正和容易混淆的其他修正有什么差异。

此字段有助于避免文档中出现大量本质重复的修正。

示例：

```markdown
- 相近修正区别：与“平局造成 1 点固定伤害”的区别是，本修正造成 1 点保底伤害，因此会受到保底伤害强化影响。
```

```markdown
- 相近修正区别：与“均衡护符”的区别是，本修正在战斗结束后按平局次数治疗；均衡护符是在槽位平局发生后立即治疗。
```

如果没有容易混淆的相近修正，可以写“无”。

---

#### 12.1.5.18 风险

风险用于记录平衡性、可读性、实现复杂度和 UI 展示问题。

示例：

- 可能削弱石头 / 剪刀 / 布胜负关系；
- 可能让战斗过慢；
- 需要较清晰的战斗日志支持；
- 可能与同名叠加后失控；
- 可能违反轮内结算阶段顺序；
- 可能需要额外 UI 展示目标或计数。

### 12.1.7 伤害源、治疗源与伤害类别规则

#### 12.1.7.1 伤害源必须明确

凡是造成伤害的修正，都应明确伤害源。

伤害源影响：

- 是否触发牌的吸血或其他伤害相关特质；
- 战斗日志如何记录；
- 后续统计中伤害归属于谁；
- 某些修正是否能识别这段伤害。

默认规则：

- 牌源伤害可以触发该牌自身的伤害相关特质；
- 非牌源伤害不触发牌特质，除非修正明确说明。

---

#### 12.1.7.2 治疗源必须明确

凡是产生治疗的修正，都应明确治疗源。

治疗源影响：

- 是否触发未来可能存在的治疗相关效果；
- 战斗日志如何记录；
- 后续统计中治疗归属于谁。

默认规则：

- 牌源治疗可以触发该牌自身的治疗相关效果；
- 非牌源治疗不触发牌特质，除非修正明确说明。

---

#### 12.1.7.3 伤害类别必须明确

凡是造成伤害的修正，都应明确伤害类别。

伤害类别影响：

- 哪些伤害强化修正可以影响它；
- 它是否属于反伤、毒伤、治疗转伤等特定构筑路线；
- 后续统计和日志如何分类。

设计时应避免为每个触发来源创建独立伤害类别。

例如，不推荐把以下内容都做成独立伤害类别：

- 平局伤害；
- 移位伤害；
- 位置 1 伤害；
- 治疗伤害。

更推荐用：

- 触发条件说明为什么发生；
- 伤害类别说明这段伤害在规则上属于哪类。

---

#### 12.1.7.4 保底伤害与固定伤害

**保底伤害** 是一种伤害类别。

例如：

```text
造成 1 点保底伤害。
```

这里的 `1` 是基础数值。如果玩家拥有“保底伤害 +1”类修正，实际伤害可以被提高。

**固定伤害** 是不归入其他伤害类别的固定数值伤害。它通常不被保底伤害、反伤、毒伤等类别强化影响。

例如：

```text
战斗开始时，敌人受到 2 点固定伤害。
```

如果设计者希望某段伤害参与某个伤害类别的联动，就不应使用固定伤害，而应明确使用对应类别。

---

### 12.1.8 轮内结算阶段兼容性检查

当前一轮结算遵循固定阶段顺序：

1. 入场阶段；
2. 固定自属性阶段；
3. 移位阶段；
4. 版面派生阶段；
5. 槽位结算阶段；
6. 合并伤害阶段；
7. 结算后阶段。

修正设计必须尊重这个顺序。

#### 12.1.8.1 禁止回头修改已结算强度

如果某个修正在槽位结果产生后触发，它不应回头修改本轮已经参与槽位结算的强度、胜负或槽位结果。

不推荐设计：

```text
槽位 1 发生槽位平局后，槽位 2 和槽位 3 的玩家牌本轮 +1 强度。
```

原因是：

- 槽位平局要到槽位结算阶段才知道；
- 本轮强度已经在版面派生阶段计算完毕；
- 如果此时再修改强度，就需要重新计算其他槽位结果；
- 这会让规则、程序和玩家理解都变混乱。

#### 12.1.8.2 推荐改写方式

如果某个设计看起来会违反阶段顺序，可以改成：

- 影响下一轮；
- 影响本轮结算后收益；
- 造成额外伤害；
- 产生治疗；
- 改变下一次奖励；
- 改变后续战斗；
- 记录本轮事件，并在未来阶段消费。

推荐改写：

```text
槽位 1 发生槽位平局后，下一轮新打出的玩家牌 +1 强度。
```

```text
槽位 1 发生槽位平局后，本轮结束时玩家回复 1 HP。
```

```text
槽位 1 发生槽位平局后，敌人受到 1 点保底伤害。
```

### 12.1.9 同名修正叠加规则

当前讨论默认：

> 同名修正可以叠加，除非该修正明确说明不叠加或有触发上限。

例如：

```text
两个均衡护符可以分别触发。
```

如果某个修正不允许叠加，应在“限制”字段中明确写出。

示例：

```markdown
- 限制：同名修正不叠加。
```

或者：

```markdown
- 限制：每轮最多触发 1 次，即使拥有多个同名修正。
```

强力修正、倍增型修正、容易导致循环触发的修正，应优先考虑添加叠加限制或触发上限。

---

### 12.1.10 重复修正整理原则

头脑风暴中应避免把本质相同的修正拆成大量重复条目。

#### 12.1.10.1 合并相近变体

如果两个修正只有轻微数值或结算类别差异，可以合并成一个修正条目，并在其中列出子方案。

例如：

```markdown
##### 平局伤害

**效果概述：**
发生槽位平局时，敌人受到少量伤害。

可选版本：
- 版本 A：造成 1 点保底伤害，可与保底伤害强化联动。
- 版本 B：造成 1 点固定伤害，不参与保底伤害强化。
```

#### 12.1.10.2 需要分开时说明区别

如果两个修正确实需要分开列举，应在“相近修正区别”字段中说明它们的设计差异。

例如：

- 即时治疗 vs 战斗结束治疗；
- 槽位源伤害 vs 牌源伤害；
- 保底伤害 vs 固定伤害；
- 影响下一轮强度 vs 造成本轮结算后伤害。

---

### 12.1.11 有序流派组合讨论法

为了避免流派组合讨论无限重复，Chat D 应使用 **有序流派组合讨论法**。D主负责规划全局顺序和专项归属，D专项只展开任务包分配给自己的组合范围。

#### 12.1.11.1 基本规则

先确定一个流派讨论顺序。

示例：

1. 平局流；
2. 移位流；
3. 位置流；
4. 邻接阵型流；
5. 单类型偏科流；
6. 三色均衡流；
7. 特质专精流；
8. 低血 / 续航 / 风险流；
9. 压制 / 控制流；
10. 爆发 / 终结流；
11. 奖励 / 经济 / 构筑操控流；
12. 敌人针对流；
13. 随机 / 混乱流。

讨论某个流派时，应讨论它与后续尚未深入讨论流派之间的混合触发修正。

后续讨论其他流派时，不再重复讨论它与已经完成流派之间的组合。

在多专项协作中：

- D主维护全局流派顺序和组合覆盖表；
- D主在任务包中明确每个 D专项负责哪些组合；
- D专项不得自行扩展到其他专项已经负责的组合；
- 遇到归属不清或跨章节冲突时，应写入完成报告并交回 D主；
- D主在合并前检查是否存在重复组合或遗漏组合。

---

#### 12.1.11.2 示例

讨论平局流时，可以讨论：

- 平局 + 移位；
- 平局 + 位置；
- 平局 + 邻接；
- 平局 + 单类型；
- 平局 + 三色均衡；
- 平局 + 特质专精；
- 平局 + 低血；
- 平局 + 压制；
- 平局 + 爆发；
- 平局 + 奖励经济；
- 平局 + 敌人针对。

之后讨论移位流时，不再重复“移位 + 平局”，只讨论：

- 移位 + 位置；
- 移位 + 邻接；
- 移位 + 单类型；
- 移位 + 三色均衡；
- 等等。

这样可以系统覆盖组合空间，同时避免重复讨论。

---

### 12.1.12 混合触发与涌现联动

#### 12.1.12.1 混合触发修正

混合触发修正是设计者主动设计出的组合修正。

例如：

```text
槽位 1 发生槽位平局时，敌人受到 1 点保底伤害。
```

它的触发条件同时包含：

- 槽位平局；
- 槽位 1。

混合触发修正需要在流派文档中系统展开。

---

#### 12.1.12.2 涌现联动

涌现联动是多个独立修正自然组合后产生的联动。

例如：

```text
修正 A：发生槽位平局时，敌人受到 1 点保底伤害。
修正 B：保底伤害 +1。
```

这两个修正同时存在时，平局造成的保底伤害会被提高。

涌现联动不需要穷尽列举。只需要记录：

- 特别有趣的组合；
- 特别危险的组合；
- 可能影响平衡或实现的组合；
- 可能让 Chat A 需要额外注意的组合。

---

### 12.1.13 D专项完成前的检查清单

D专项准备把修正写入最终章节或完成报告前，建议至少检查以下问题。

#### 12.1.13.1 规则清晰性

- 效果概述是否能让人快速理解？
- 触发条件是否明确？
- 作用时机是否明确？
- 作用对象类型是否明确？
- 作用范围和筛选条件是否足够清楚？

#### 12.1.13.2 战斗系统兼容性

- 是否违反当前轮内结算阶段顺序？
- 是否要求回头修改本轮已经结算过的强度或槽位结果？
- 是否需要额外事件记录或计数器？
- 是否需要新的 UI 展示？

#### 12.1.13.3 伤害与治疗严谨性

如果修正造成伤害：

- 是否写明伤害源？
- 是否写明伤害类别？
- 是否会触发牌特质？
- 是否会参与某类伤害强化？

如果修正产生治疗：

- 是否写明治疗源？
- 是否受最大 HP 限制？
- 是否可能触发治疗相关联动？

#### 12.1.13.4 流派价值

- 它是否改变玩家评价局面的方式？
- 它是否提供了新的构筑路线？
- 它是否只是单纯数值增强？
- 它是否与已有流派有清晰区分？

#### 12.1.13.5 平衡与复杂度风险

- 是否可能过强？
- 是否可能拖慢战斗？
- 是否需要复杂 UI 才能理解？
- 是否和同名叠加后失控？
- 是否和某些涌现联动过强？

---

### 12.1.14 D主移交 A主前的检查

D主不需要把所有 Chat D 候选一次性交给 A主。准备 handoff 时，应先完成以下整理：

- 确认候选所属流派和主要归属章节；
- 合并只有轻微数值或类别差异的重复变体；
- 保留确有不同构筑身份的相近方案，并说明区别；
- 区分已确认机制结构与尚未平衡数值；
- 标记需要新状态、新 UI、新阶段或跨战斗保存的候选；
- 标记规则改写型、循环型、倍增型和可能无限触发的高风险候选；
- 汇总最适合早期版本、最有长期构筑价值和最适合角色核心被动的候选；
- 明确希望 A主作出的决策，例如进入版本、延期、合并、删除或要求补充讨论。

推荐的 D主 → A主 handoff 至少包含：

```text
候选来源：
<流派文档、章节和内容版本>

候选范围：
<本次希望 A主评估的范围>

优先候选：
- <候选 1>
- <候选 2>

已确认机制结构：
- <结构结论>

尚未平衡问题：
- <数值、稀有度、概率、叠加等>

高风险或高成本项：
- <规则、UI、状态、工程风险>

希望 A主决定：
- <进入近期版本 / 延期 / 合并 / 删除 / 继续讨论>
```

### 12.1.15 A主评估结果与文档去向

A主评估后，应明确候选的去向：

```text
接受进入正式玩法范围
→ 更新对应 canonical gameplay 文档；
  必要时新增或更新 GDR。

保留但不进入近期版本
→ 保留在流派候选文档，或移入 deferred-gameplay-ideas.md。

需要进一步探索
→ 返回 D主，由 D主创建补充 D专项或继续现有专项。

机制可用但过于复杂
→ 由 A主给出简化目标，再返回 D主或进入正式设计细化。

拒绝
→ 从活跃候选池移除；必要时保留简短拒绝理由，避免重复讨论。
```

当本节的方法论本身发生变化时：

- 修正设计轴、模板、检查清单和组合覆盖方法 → 更新本节；
- Chat 角色、D主/D专项、handoff、恢复或共同基线规则 → 更新本文档对应的 `[Shared]` 或角色专属章节；
- 术语定义 → 更新 `Docs/game-design/glossary.md`；
- 正式玩法规则 → 更新对应 canonical gameplay 文档或 GDR。



# 13. 文档维护规则 `[Shared]`

### 13.1 只保存稳定流程和当前快照

本文档应保存：

- 已稳定的协作方式；
- handoff 模板；
- 恢复流程；
- 跨 Chat 共用的稳定设计与评估方法；
- 当前简短状态。

不应保存：

- 每次聊天的完整过程；
- 大量历史任务日志；
- 已经过期的每日状态；
- 可从 canonical 文档直接获取的规则全文。

### 13.2 谁负责更新哪一部分

```text
[Shared]
由相关主协调会话共同维护。

[Chat A Only]
由 Chat A 补充和维护。

[Chat B Only]
由 Chat B 补充和维护。

[Chat C Only]
由 Chat C 补充和维护。

[Chat D Only]
由 Chat D 补充和维护。

[Chat D + Chat A Only]
由 Chat D 与 Chat A 共同维护；方法论由 D主整理，评估与正式化边界由 A主确认。
```

其他角色不应静默改写某个角色的专属工作协议。

### 13.3 更新时机

建议在以下时机更新：

- 工作方式发生明显变化；
- 新的主协调 / 专项结构被采用；
- 新的 handoff 模板被确认；
- 某个 Chat 角色总结出稳定沟通规则；
- 当前主会话需要恢复；
- 一个流派或实现阶段完成；
- 当前状态快照明显过期。

### 13.4 文档目标

当任意主会话失效时，用户应能依靠：

```text
Docs/project-workflow-and-handoff.md
+ 当前 source-of-truth 文档
+ 当前累计文档
+ 当前状态快照
```

恢复新的主协调会话，并继续工作，而不需要重新摸索协作方式。
